# Plan Verification

입력 문서(설계서/구현계획서) 기반 구현 점검 스킬. 멀티 에이전트 병렬 점검 → 수정 → 재검증 루프를 실행한다.

**실패 비용:** false negative(문제를 놓침)가 false positive(멀쩡한 것을 지적)보다 비싸다. 놓친 불일치는 운영에서 장애로 나타나지만, 잘못된 지적은 수정 단계에서 "이건 아니다"로 걸러진다. 의심스러우면 FAIL로 보고한다.

## When to Use

- 구현계획서 또는 설계서가 존재하고 코드가 작성된 후
- "플랜대로 됐는지 확인해줘", "점검해줘", "리뷰하고 수정해줘"

## When NOT to Use

- 플랜 없이 코드만 리뷰 → `review` 스킬
- 버그 원인 추적 → `investigate` 스킬
- 프롬프트만 단독 점검 → `prompt-golden-rules` 직접 사용
- 단일 파일/함수 수준 수정 → 에이전트 팀 불필요

## 워크플로우

```
INPUT: 사용자 요청
    ▼
Phase 0: 입력 문서 특정 — 경로 있으면 바로 진행, 없으면 탐색 → 선택
    ▼
Phase 1: 분석 — 입력 문서에서 카테고리 도출 + 점검 항목 생성
    ▼
Phase 2: 점검 — 카테고리별 에이전트 병렬 디스패치
    ▼
Phase 3: 종합 — 보고서 병합 → FAIL/REVIEW 분류
    ▼
FAIL 있나? ──No──→ Phase 5: 최종 보고
    │
   Yes
    ▼
Phase 4: 수정 — 오케스트레이터가 FAIL 수정 → 검증 실행
    ▼
Phase 2로 복귀 (전체 재검증)
```

## Phase 0: 입력 문서 특정

사용자가 입력 문서 경로를 명시하면 바로 Phase 1로 진행한다. 경로가 없으면 아래 흐름으로 입력 문서를 탐색하여 선택한다.

**입력 분류:**

| 사용자 입력 | 동작 |
|------------|------|
| 파일 경로 (`docs/plans/xxx.md`, `~/.claude/plans/xxx.md`) | Phase 1로 직행 |
| 키워드/주제 ("위조 검증 플랜", "extraction 점검") | 탐색 → 후보 제시 → 선택 |
| 구체적 정보 없음 ("플랜 점검해줘") | 주제를 물어본 뒤 탐색 |

**탐색 범위:** 프로젝트의 설계서/구현계획서가 있는 위치를 탐색한다.
- `docs/plans/**/*.md` — 프로젝트 설계서/구현계획서
- `~/.claude/plans/*.md` — Claude 세션 구현계획서

**탐색 방법:**
1. 사용자가 키워드를 제공한 경우: 위 경로들에서 Glob으로 파일 목록을 가져온 후, 파일명과 내용 첫 줄(제목)을 키워드와 대조하여 관련 파일을 필터링한다
2. 키워드가 모호하면 Grep으로 파일 내용까지 검색한다
3. 사용자가 키워드 없이 "플랜 점검"만 요청한 경우: AskUserQuestion으로 주제를 먼저 물어본다

**후보 제시 — 복수 선택 지원:**

관련 문서가 발견되면 AskUserQuestion(`multiSelect: true`)으로 옵션을 제시한다. 각 옵션에는 파일명과 제목(첫 번째 `#` 헤더)을 표시한다. 0개면 사용자에게 경로를 직접 입력받는다.

같은 기능에 대해 설계서와 구현계획서가 쌍으로 존재하는 경우가 흔하다 (예: `*-design.md` + `*-implementation-plan.md`). 복수 선택하면 Phase 1에서 각 문서의 역할을 분류하여 점검 품질을 높인다:

| 입력 문서 조합 | Phase 1 활용 |
|---------------|-------------|
| **설계서 + 구현계획서** | 설계서 → 설계 원칙/의도(WHY) 추출, 구현계획서 → 코드 레벨 체크리스트(HOW) 추출. 최상 품질 |
| **구현계획서만** | 코드 정합성, 테스트 완전성, 프롬프트 품질 점검. 설계 의도 검증은 제한적 |
| **설계서만** | 설계 원칙, 아키텍처 방향 점검 가능. 코드 레벨 정합성은 설계서에 구체적 코드 명세가 있는 범위까지만 |

**문서 역할 자동 분류:** 파일명 패턴(`design`, `설계`) 또는 문서 내용(구현 단계/Step 유무, 코드 블록 유무)으로 설계서/구현계획서를 구분한다. 구분이 모호하면 사용자에게 확인한다.

**예시 흐름:**

사용자: "위조 검증 플랜 점검해줘"
→ `docs/plans/` 에서 "위조", "fraud", "검증" 키워드로 탐색
→ 후보 발견 2건 — AskUserQuestion(multiSelect: true):
  - `2026-04-05-resume-fraud-detection-design.md` — 이력서 위조 탐지 설계
  - `2026-04-05-resume-fraud-detection-implementation-plan.md` — 이력서 위조 탐지 구현 계획
→ 사용자가 둘 다 선택
→ Phase 1에서 설계서(의도/원칙) + 구현계획서(코드 체크리스트) 분류 후 진행

## Phase 1: 분석

오케스트레이터가 Phase 0에서 선택된 문서를 읽고, **변경 유형**에 따라 점검 카테고리를 도출한다.

**복수 문서 처리:** 설계서와 구현계획서가 함께 입력된 경우, 각 문서에서 역할에 맞는 정보를 추출한다:
- **설계서** → 설계 원칙, 아키텍처 결정 이유, 알려진 한계, 의도된 동작 → 주로 "설계 원칙 준수" 카테고리에 활용
- **구현계획서** → Step별 코드 변경, 테스트 정의, 검증 커맨드 → 주로 "코드 정합성", "테스트 완전성", "실행 검증" 카테고리에 활용

두 문서에서 중복되는 정보는 병합한다. 충돌하는 정보는 사용자에게 확인한다.

**카테고리 도출 원칙:** 문서에서 "어떤 파일에 무엇이 있어야 하는가"를 서술하는 모든 구절을 점검 항목으로 변환한다. 단계(Step) 구조면 단계별로, 서술형이면 문장 단위로 점검 항목을 추출한다.

| 카테고리 | 생성 조건 | 주요 소스 |
|---------|----------|----------|
| **코드 정합성** | 항상 | 구현계획서 |
| **테스트 완전성** | 문서에 테스트 정의가 있을 때 | 구현계획서 |
| **실행 검증** | 항상 (문서의 "검증 방법" 또는 프로젝트 설정에서 커맨드 추출) | 구현계획서 / 프로젝트 설정 |
| **설계 원칙 준수** | 문서에 설계 원칙/알려진 한계 섹션이 있을 때 | 설계서 |
| **LLM 프롬프트 품질** | 수정 대상에 LLM 프롬프트가 포함될 때 | 구현계획서 |

입력 문서에 위 표에 없는 변경 유형이 있으면 해당 카테고리를 추가한다. 해당 유형이 없으면 생성하지 않는다.

**정보 보존:** 에이전트에게 점검 항목 목록과 함께 **입력 문서 원본 경로**를 전달한다. 이유: 점검 항목 압축 시 맥락이 소실될 수 있으므로, 판단이 필요할 때 원문을 참조할 수 있어야 한다.

## Phase 2: 점검 — 에이전트 디스패치

카테고리별로 필요한 에이전트만 병렬 디스패치한다. 카테고리가 없으면 해당 에이전트는 생성하지 않는다.

### 에이전트별 프롬프트 필수 요소

모든 에이전트 프롬프트에는 아래 6가지를 포함한다. 이유: RED 테스트에서 이 요소가 빠지면 에이전트가 코드 리딩만 하고 종료하거나, 프레임워크 없이 자유롭게 분석하여 구조적 품질 이슈를 놓쳤다.

| 요소 | 내용 |
|------|------|
| **목적 (WHY)** | 이 에이전트가 왜 존재하는가. 점검하지 않으면 어떤 문제가 발생하는가 |
| **입력 (WHAT)** | 읽어야 할 파일 경로, 체크리스트 항목 |
| **판단 기준 (HOW)** | PASS/FAIL을 어떻게 판정하는가. 원칙으로 서술 |
| **실패 비용** | false negative > false positive. 의심스러우면 FAIL |
| **경계 사례** | PASS/FAIL 경계 1~2쌍 |
| **출력 형식** | 이슈별 보고서: 각 이슈에 `[카테고리-번호]`, 대상 파일 경로, 심각도(critical/minor), PASS/FAIL 판정, 근거를 포함. Phase 3에서 오케스트레이터가 에이전트 간 교차 대조할 수 있도록 파일 경로 기준으로 정렬 |

### Code Reviewer (코드 정합성)

입력 문서와 실제 코드를 대조하여 **의미적 불일치**를 식별한다. 놓친 불일치는 운영에서 발견될 때까지 잠복한다.

**심각도:** critical(기능 동작 영향) / minor(기능 무관, 입력 문서와 불일치)

**경계 사례:**
- `["admin", "editor", "viewer"]` vs `["admin", "viewer", "editor"]` → **PASS** (순서 차이, 의미 동일)
- `["admin", "editor", "viewer"]` vs `["admin", "editor"]` → **FAIL** (항목 누락, 기능 차이)

### Test Verifier (테스트 완전성 + 실행 검증)

테스트 완전성 확인 + **실제 테스트/린트 실행**. 코드 리딩만으로는 런타임 실패(환경 의존, 타이밍, 데이터 상태)를 발견할 수 없으므로 테스트 커맨드를 실행해야 한다. 실행이 불가능한 경우(외부 서비스 의존, CI 전용 테스트 등)는 실행 불가 사유를 보고서에 명시하고 코드 리딩 기반 점검으로 전환한다.

**경계 사례:**
- 플랜 "4개 레코드 생성" / 코드 "3개 + 1개 제외 대상" → **PASS** (검증 논리 동일)
- 플랜 "입력값 변형 테스트" 요구 / 코드에 해당 테스트 없음 → **FAIL** (커버리지 누락)

### Prompt Quality Reviewer (LLM 프롬프트 품질)

**REQUIRED SUB-SKILL:** `prompt-golden-rules` 스킬을 사용하여 점검한다.

RED 테스트에서 에이전트가 골든룰 프레임워크 없이 프롬프트를 분석하면, 논리적 모순은 찾지만 구조적 품질 이슈(WHY 부재, 하드코딩 패턴)를 놓쳤다. 이 에이전트는 반드시 골든룰 3원칙으로 점검해야 한다.

**프롬프트 식별:** 수정 대상 파일에서 LLM API 호출 함수를 찾고, 인자로 전달되는 문자열을 역추적한다.

**수정안에 사례를 추가할 때 — 웹검색으로 도메인 맥락 보강:**
프롬프트가 도메인 특화 용어/분류를 다루고 에이전트가 해당 도메인의 관행을 확신할 수 없을 때, 웹검색으로 업계 실사용 패턴을 확인한 후 사례를 작성한다. 이유: 도메인 지식 없이 사례를 지어내면 LLM이 잘못된 경계를 학습한다.

**경계 사례:**
- "카테고리는 category_tags에, 개별 항목은 item_keywords에" → **PASS** (판단 기준 명확)
- "category_tags와 item_keywords를 동시에 사용하지 마세요" → **FAIL** (WHY 없음)

### Design Principle Verifier (설계 원칙 준수)

설계 원칙이 코드와 테스트에서 실제로 보장되는지 확인한다. 원칙 위반은 특정 입력에서만 드러나는 조용한 버그를 만든다.

**경계 사례:**
- "대소문자 무시" 원칙에 대해 해당 테스트 존재 → **PASS**
- "미인식 입력 폴백 처리" 원칙에 대해 해당 테스트 없음 → **FAIL**

## Phase 3: 종합

오케스트레이터가 에이전트 보고서들을 수집하고:

1. 전체 판정 결과를 **PASS / FAIL / REVIEW**로 분류
2. **REVIEW 판정:** 에이전트가 FAIL로 보고했으나, 해당 기능을 직접 커버하는 테스트가 존재하고 통과하며 설계 원칙도 충족하는 경우 → REVIEW로 재분류. 이유: 테스트가 해당 동작을 검증하고 있다면 입력 문서 자체의 오류일 수 있으므로 자동 수정 대신 사용자 판단에 위임. 단, 해당 기능의 테스트가 없는 상태에서 단순히 "기존 테스트가 깨지지 않았다"는 것만으로는 REVIEW로 재분류하지 않는다
3. FAIL이 0개면 Phase 5 (REVIEW만 남아도 Phase 5)
4. FAIL이 있으면 심각도 순(critical → minor)으로 Phase 4. 이유: critical은 다른 수정의 전제 조건이 될 수 있다

## Phase 4: 수정

**수정 주체:** 오케스트레이터가 직접 수정한다. 이유: 에이전트들의 보고서를 종합하여 충돌을 해결해야 하므로 전체 맥락을 가진 오케스트레이터가 수정해야 한다. 수정 후 Phase 2에서 에이전트를 새로 디스패치하므로 "자기 수정 자기 검증" 문제는 발생하지 않는다.

**수정 규칙:**
- 입력 문서가 기준이다. 이유: 입력 문서는 사용자가 의도적으로 설계한 사양이므로, 의도 없는 이탈은 버그다
- REVIEW 판정 이슈는 수정하지 않고 최종 보고에 포함
- 프롬프트 품질 FAIL은 golden rules 수정안을 적용
- 수정 범위는 입력 문서에서 명시한 변경 대상 파일로 제한. 이유: 입력 문서 범위 밖 변경은 별도 위험이 있다
- 수정 후 검증 커맨드 실행

수정 완료 후 Phase 2로 복귀하여 **전체 재검증**한다. 이유: 수정이 다른 항목에 부작용을 일으킬 수 있다.

## 종료 조건

- 전 이슈 PASS → 즉시 종료
- 이번 반복에서 새로 PASS된 이슈가 0개 → 진전 없음, 사용자 에스컬레이션

진전이 있는 한 반복을 계속한다.

**에스컬레이션:**
- 같은 이슈가 2회 연속 FAIL이고 수정 시도가 동일 → 사용자에게 에스컬레이션
- 수정이 다른 이슈를 FAIL로 만드는 경우(seesaw) → 충돌 이슈 명시, 사용자 판단 요청

## Phase 5: 최종 보고

```
## 점검 결과 요약

| 카테고리 | 전체 | PASS | FAIL | REVIEW | 수정됨 |
|----------|------|------|------|--------|--------|
| {카테고리명} | N | N | N | N | N |

## REVIEW 항목 (사용자 판단 필요)
```
