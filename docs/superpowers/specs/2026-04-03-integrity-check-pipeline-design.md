# 이력서 위조 탐지 파이프라인 설계

## 문제

현재 이력서 검수 시스템은 규칙 기반(discrepancy.py)으로 날짜 산술 오류만 탐지한다.
사용자가 원하는 것은 **의도적 사기(fraud)** 탐지 — 경력 위조, 학력 세탁, 기간 조작 등
채용 담당자를 속이려는 의도적 행위의 흔적을 찾는 것이다.

규칙 기반으로는 "국문에 1999년, 일문에 1992년"이나 "2개월 재직인데 11개월로 기재" 같은
원문 맥락 기반의 모순을 잡을 수 없다. LLM이 원문을 읽는 시점에서만 탐지 가능하다.

## 설계 원칙

1. **태스크 마이크로 분해**: 각 단계가 하나의 명확한 역할만 수행
2. **정보 소실 금지**: 추출 단계에서 데이터를 버리지 않음
3. **프롬프트 황금률**: 하드코딩 금지 + 충분한 맥락 + 최적화 사례의 균형
4. **단계 간 피드백 루프**: 뒷 단계가 앞 단계의 품질을 검증하고, 문제 시 자동 재시도

## 아키텍처 개요

```
┌─────────────────────────────────────────────────────┐
│ 이력서 원본 (.doc/.docx)                              │
│   ↓ 텍스트 추출 + 전처리 (기존)                        │
│   ↓                                                  │
│ Step 1: 충실한 추출 (Gemini 1회)                       │
│   모든 섹션, 모든 언어에서 중복 제거 없이 전부 추출        │
│   ↓ raw JSON (N개 항목, source_section 포함)           │
│                                                      │
│ Step 1.5: 시멘틱 그룹핑 (Gemini 1회, 경량)              │
│   동일 회사/학교 매칭 (다국어, 전체-세부 관계 포함)         │
│   ↓ 그룹화된 JSON                                     │
│                                                      │
│ Step 2: 그룹별 정규화 + 위조 탐지 (Gemini 병렬, 그룹당 1회)│
│   각 에이전트: 그룹 내 통합 + DATE_CONFLICT 탐지          │
│   학력 에이전트: 통합 + SHORT_DEGREE 탐지                │
│   ↓ 정규화된 careers[], educations[], integrity_flags[] │
│                                                      │
│ Step 3: 교차 분석 (코드, LLM 불필요)                     │
│   정규화된 경력 간 PERIOD_OVERLAP 계산                   │
│   결과 병합                                            │
│   ↓ 최종 Candidate 데이터 + integrity_flags             │
│                                                      │
│ * 각 단계에 품질 검증 + 자동 재시도 루프 내장             │
│   (기존 retry_pipeline.py 패턴 확장)                    │
└─────────────────────────────────────────────────────┘
```

## Step 1: 충실한 추출

### 역할
이력서 원문 텍스트에서 **모든 데이터를 있는 그대로** 가져온다.
판단, 선택, 정규화를 하지 않는다.

### 왜 이렇게 하는가
기존 파이프라인은 추출과 정규화를 동시에 수행하여, 섹션 간 날짜 충돌이
정규화 과정에서 소실되었다. 이 단계를 분리하면 정보 소실 없이 모든 원문
데이터가 보존된다.

### 프롬프트 설계 원칙

**섹션별 독립 추출**: 같은 회사/학교가 여러 섹션이나 여러 언어에 나오면
각각 별도 항목으로 추출한다. source_section으로 출처를 구분한다.
이렇게 하는 이유: 정규화 시스템이 섹션 간 날짜를 비교해야 하기 때문이다.

**서술문 포함**: 구조화된 테이블뿐 아니라, 서술문에서 구체적 기관명이 포함된
경력/학력 언급도 추출한다.

**원문 보존**: 날짜, 기간 표기, 기관명을 원문 그대로 가져온다.
정규화는 다음 단계의 역할이다.

**부가 정보 보존**: 경력 항목에 괄호로 기간이 표기되어 있으면 duration_text에 가져온다.
이 정보는 다음 단계에서 날짜와 기간의 정합성을 검증하는 데 사용된다.
시작~종료일과 기재된 기간이 모순되면 위조 의심 신호이기 때문이다.

### 실패 비용 구조
이 단계에서의 **누락은 위조 탐지 불가**로 이어진다 — 한 섹션의 데이터가 빠지면
섹션 간 비교가 불가능하다. 반면 **과잉 추출은 다음 단계에서 걸러진다** —
불필요한 항목이 포함되어도 정규화 단계에서 통합되므로 최종 결과에 영향이 작다.
따라서 누락보다 과잉이 훨씬 안전하다.

### 입력
전처리된 이력서 텍스트

### 출력 스키마
```json
{
  "name": "string",
  "name_en": "string | null",
  "birth_year": "integer | null",
  "email": "string | null",
  "phone": "string | null",
  "total_experience_text": "string | null",
  "resume_reference_date": "string | null",
  "careers": [
    {
      "company": "string (원문 그대로)",
      "position": "string | null",
      "start_date": "string | null (원문 그대로)",
      "end_date": "string | null (원문 그대로)",
      "duration_text": "string | null",
      "is_current": "boolean",
      "duties": "string | null",
      "source_section": "string"
    }
  ],
  "educations": [
    {
      "institution": "string (원문 그대로)",
      "degree": "string | null",
      "major": "string | null",
      "start_year": "integer | null",
      "end_year": "integer | null",
      "status": "string | null (졸업/중퇴/수료 등)",
      "source_section": "string"
    }
  ],
  "certifications": [...],
  "language_skills": [...]
}
```

### 검증 기준
- 이력서의 모든 언어 섹션에서 데이터가 추출되었는가
- 같은 회사가 N개 섹션에 나오면 N개 항목이 있는가
- 서술문 내 학력/경력 언급이 포함되었는가
- duration_text가 원문 괄호 표기 그대로인가

## Step 1.5: 시멘틱 그룹핑

### 역할
Step 1의 raw 항목들을 **동일 회사/학교** 단위로 그룹핑한다.
병합은 하지 않는다. 누가 누구와 같은 항목인지만 판별한다.

### 왜 필요한가
Step 2의 각 에이전트가 하나의 그룹만 처리하려면, 사전에 그룹핑이 되어 있어야 한다.
다국어 매칭과 전체-세부 관계 파악은 단순 문자열 비교로 불가능하므로
LLM의 시멘틱 이해가 필요하다.

### 왜 여기서 병합하지 않는가
그룹핑과 병합을 동시에 하면 태스크가 복합적이 된다.
"같은 회사인가?"라는 판단과 "어떤 날짜를 선택할 것인가?"라는 판단은 별개이다.
그룹핑만 하면 태스크가 단순하고, 잘못된 그룹핑을 병합 전에 확인할 수 있다.

### 그룹핑 유형
- 동일 회사가 다른 언어로 표기된 경우 (시멘틱 매칭 필요)
- 한 회사의 전체 기간과 세부 직무 기간이 함께 있는 경우 (날짜 포함 관계)
- 계열사/그룹사 관계 (시멘틱 이해 필요)
- 동일 회사의 같은 언어 반복 (문자열 매칭으로 가능)

### 실패 비용 구조
- **잘못 나누기** (같은 회사를 다른 그룹으로): Step 2에서 별개 회사로 처리되어
  날짜 충돌을 발견할 수 없음. 심각.
- **잘못 묶기** (다른 회사를 같은 그룹으로): Step 2에서 잘못된 통합 시도.
  발견 가능하지만 비효율.
- **미분류**: 확실하지 않은 경우 그룹에 넣지 않는 것이 잘못 묶는 것보다 안전.

### 프롬프트 설계 원칙
확실하지 않은 경우 그룹에 넣지 않고 미분류(ungrouped)로 남긴다.
잘못 묶는 것보다 안 묶는 것이 안전하기 때문이다.
미분류 항목은 Step 2에서 개별 처리된다.

### 입력
Step 1의 raw JSON

### 출력 스키마
```json
{
  "career_groups": [
    {
      "group_id": "string",
      "canonical_name": "string (대표 회사명, 한국어 우선)",
      "relationship": "string (same_company | parent_with_sub_periods | affiliated_group)",
      "entry_indices": [0, 4, 7, 11]
    }
  ],
  "education_groups": [
    {
      "group_id": "string",
      "canonical_name": "string",
      "entry_indices": [0, 3]
    }
  ],
  "ungrouped_career_indices": [],
  "ungrouped_education_indices": []
}
```

### 구현 옵션
- **Option A**: Gemini 1회 호출 (경량, 전체 그룹핑)
- **Option B**: 코드(문자열 유사도 + 날짜 포함 관계) + LLM(다국어/계열사만)
- 권장: Option A (단순성, 그룹핑 태스크 자체가 경량)

## Step 2: 그룹별 정규화 + 위조 탐지

### 역할
각 그룹의 항목들을 **하나의 정규화된 레코드**로 통합하고,
통합 과정에서 발견되는 **해소 불가능한 모순**을 integrity_flags로 기록한다.

위조 탐지는 별도의 작업이 아니라 **정규화의 부산물**이다.
통합이 매끄럽게 되면 정상, 통합 과정에서 해소할 수 없는 모순이 발견되면 위조 의심.

### 왜 병렬인가
각 그룹은 독립적이다. 카모스테크의 날짜 충돌 판단에 삼성생명 데이터는 불필요하다.
따라서 그룹별 에이전트를 병렬로 실행하면:
- 각 에이전트의 입력이 작아 판단 품질 향상
- 전체 처리 시간 단축
- 한 그룹의 실패가 다른 그룹에 영향 없음

### 프롬프트 설계 원칙

**출력 용도**: 정규화된 데이터는 후보자 DB에 저장되어 검색·열람에 사용된다.
integrity_flags는 채용 담당자에게 검수 알림으로 표시된다.

**날짜 선택**: 이력서의 서로 다른 섹션은 서로 다른 시점에 작성되었을 수 있다.
세부 기간에 "현재"가 남아 있어도 전체 기간에 확정 종료일이 있다면,
전체 기간이 이후에 업데이트된 최종 정보일 가능성이 높다.
따라서 가장 확정적인 정보를 선택한다.

**위조 탐지 프레이밍**: "통합이 매끄러우면 빈 배열, 안 되면 flag."
별도로 "위조를 찾아라"가 아니라, 정규화를 하다가 안 풀리는 것이 위조 신호.

**실패 비용**: 거짓 경보(false positive)는 채용 담당자의 시스템 신뢰를 떨어뜨린다.
담당자가 RED 알림을 보면 해당 후보자를 즉시 재검토하고,
YELLOW를 보면 면접 시 확인 사항으로 기록한다.
따라서 RED는 다른 합리적 설명이 정말로 어려운 경우에만 사용한다.

### 경력 그룹 에이전트

**입력**: 하나의 경력 그룹 (같은 회사의 여러 항목)

**태스크**:
1. 여러 항목을 하나의 레코드로 통합
2. 날짜 충돌 시 가장 확정적인 값 선택
3. 통합 과정에서 해소 불가능한 모순 발견 시 DATE_CONFLICT flag

**DATE_CONFLICT 판단 원칙**:
- 통합 시 차이가 있었지만 타이핑 실수나 월 계산 방식 차이로
  설명할 수 있으면 정규화만 하고 보고하지 않음
- 설명할 수 없는 수준의 차이는 보고
- duration_text와 날짜 계산의 모순은 같은 항목 내부의 자기모순이므로
  정규화로 해소할 수 없음 — 보고 대상

**출력**:
```json
{
  "career": { "company": "...", "start_date": "YYYY-MM", "end_date": "YYYY-MM", ... },
  "flags": [
    {
      "type": "DATE_CONFLICT",
      "severity": "RED | YELLOW",
      "detail": "...",
      "chosen": "...",
      "alternative": "...",
      "reasoning": "..."
    }
  ]
}
```

### 학력 에이전트

**입력**: 전체 학력 항목 (그룹핑 결과 포함)

**태스크**:
1. 같은 학교의 여러 항목을 통합
2. SHORT_DEGREE 탐지

**SHORT_DEGREE 판단 원칙**:
- 본인이 솔직하게 밝힌 학력 사항은 위조가 아니다
- 같은 분야의 다른 학교가 있으면 편입 가능성이 있다
- 위 어디에도 해당하지 않으면서 정규 학위를 수업연한보다
  현저히 짧은 기간에 취득한 것은 보고 대상이다

### severity 기준

채용 담당자의 행동 기준으로 정의:
- **RED**: 담당자가 이 후보자의 이력서 신뢰성을 재검토해야 하는 수준.
  타이핑 실수, 서식 차이, 정상적 겸업 등으로 설명할 수 없는 경우.
- **YELLOW**: 담당자가 면접 시 확인 사항으로 기록하는 수준.
  의심은 가지만 정상적 설명도 가능한 경우.

## Step 3: 교차 분석 (코드)

### 역할
정규화 완료된 경력 레코드들 간의 **PERIOD_OVERLAP**을 계산한다.
LLM 불필요 — 날짜 비교 연산만.

### 기존 코드 활용
`candidates/services/discrepancy.py`의 `check_career_overlaps()`를 기반으로 하되:
- `is_current=True` 경력도 포함 (오늘 날짜를 종료일로 사용)
- 같은 회사 내 이동은 Step 1.5에서 이미 그룹핑되어 제외됨

### PERIOD_OVERLAP severity (규칙 기반)

Step 1.5의 그룹핑 정보를 활용하여 규칙으로 판정:
- 같은 그룹(affiliated_group) 간 중복 → 정상 (flag 안 함)
- 그 외 중복 → 겹침 기간과 맥락으로 severity 결정:
  - 이직 인수인계 수준의 짧은 중복 → 정상 (flag 안 함)
  - 이를 넘어서는 장기 중복 → YELLOW
  - 장기 중복이 여러 건 반복 → RED

### 결과 병합
Step 2의 그룹별 결과 + Step 3의 교차 분석을 하나의 최종 결과로 병합:
```json
{
  "careers": [...],
  "educations": [...],
  "integrity_flags": [...],
  "field_confidences": {...}
}
```

## 단계 간 피드백 루프

### 왜 필요한가

기존 `retry_pipeline.py`에 추출→검증→판정 패턴이 있지만,
실제 재시도는 하지 않고 `human_review`로 표시만 한다.
새 파이프라인에서는 이 패턴을 확장하여,
사람에게 넘기기 전에 **피드백 포함 자동 재시도**를 먼저 시도한다.

### 기존 검증과의 관계

기존 `validation.py`는 추출된 JSON의 형식적 정합성을 검증한다
(필드 존재 여부, 날짜 형식, 파일명 교차 검증). 이 구조(규칙 기반 체크 → 점수 → pass/fail)는
재활용하되, 새 파이프라인의 각 단계에 맞는 **맞춤 검증 항목**을 설계한다.

### Step 1 검증: 추출 완전성

Step 1의 역할은 "모든 데이터를 빠짐없이 가져오는 것"이므로,
검증도 **완전성**을 본다.

**코드 기반 검증 (LLM 불필요):**
- **다국어 섹션 커버리지**: 원문 텍스트에서 언어별 문자 비율을 감지하고,
  추출된 항목의 source_section에 해당 언어가 있는지 비교.
  원문에 일본어 문자가 일정 비율 이상인데 일문 source_section이 없으면 누락 의심.
- **경력/학력 수량 적정성**: 원문 텍스트의 길이 대비 추출된 항목 수가
  지나치게 적으면 누락 의심. 반대로 과도하게 많으면 노이즈 의심.
- **duration_text 채워짐 비율**: 원문에 괄호 안 기간 표기가 있는데
  duration_text가 비어 있는 항목이 있으면 누락.
- **source_section 다양성**: 모든 항목의 source_section이 동일하면
  하나의 섹션에서만 추출한 것 — 다른 섹션 누락 가능성.

**검증 실패 시**: Step 1을 피드백과 함께 재실행.
피드백에는 구체적으로 무엇이 부족한지 포함 (예: "일문 섹션 데이터가 없습니다").

### Step 1.5 검증: 그룹핑 정합성

**코드 기반 검증:**
- **ungrouped 비율**: 전체 항목 대비 미분류 비율이 높으면
  그룹핑이 제대로 안 되었거나 Step 1 추출이 부실한 것.
- **그룹 내 언어 다양성**: 다국어 이력서인데 한 그룹에 하나의 언어만 있으면
  다른 언어의 같은 회사 항목이 별도 그룹이 되었거나 ungrouped된 것.
- **그룹 내 날짜 범위 정합성**: parent_with_sub_periods 관계인 그룹에서
  세부 기간이 전체 기간을 벗어나면 잘못 묶인 것.

**검증 실패 시**: Step 1.5를 피드백과 함께 재실행.
ungrouped 비율이 높은 경우 Step 1까지 거슬러 재실행할 수 있음.

### Step 2 검증: 정규화 품질

**코드 기반 검증:**
- **필수 필드 완전성**: 정규화 결과에 company, start_date 등 필수 필드 누락 여부.
- **날짜 형식 정규화**: 모든 날짜가 YYYY-MM 형식으로 변환되었는지.
- **정보 보존**: 입력 그룹의 항목 수 대비 출력의 정보량이 적절한지.
  4개 항목이 입력됐는데 duties가 전부 null이면 정보 손실 의심.
- **flag 일관성**: severity가 있으면 chosen/alternative/reasoning도 있어야 함.

**검증 실패 시**: 해당 그룹만 재실행.

### 재시도 정책

- 각 단계 최대 1회 재시도 (무한 루프 방지)
- 재시도에도 실패하면 `human_review`로 표시 (기존 패턴 유지)
- 재시도 시 이전 실행의 출력과 문제점을 피드백으로 포함
- 재시도 이력을 `ValidationDiagnosis` 모델에 기록 (기존 모델 활용)

### 자체 개선 메커니즘

**자동 (사람 불필요):**
- 단계 간 피드백 루프 (위 설명)
- 품질 메트릭 자동 추적: 필드 채워짐 비율, ungrouped 비율, flag 발생률
- 메트릭 이상 시 알림 (프롬프트나 입력 데이터 변화 감지)

**반자동 (사람의 판단을 시스템이 수집):**
- 채용 담당자가 flag를 확인/기각할 때 결과 기록
- 축적된 판단 패턴을 주기적으로 프롬프트 개선에 반영
- DiscrepancyReport에 `reviewer_verdict` 필드 추가하여
  담당자의 확인/기각 이력 저장

**수동 (사람이 직접):**
- 새로운 위조 유형 발견 시 프롬프트에 반영
- severity 기준 재조정
- 프롬프트 황금률 기반 주기적 검토

## 기존 시스템과의 관계

### 대체하는 것
- `gemini_extraction.extract_candidate_data()` → Step 1 + 1.5 + 2로 대체
- `discrepancy.py`의 일부 체크 → Step 2 + 3이 대체

### 유지하는 것
- `text_extraction.py` — 텍스트 추출 + 전처리 (변경 없음)
- `validation.py` — 추출 후 규칙 검증 (Step 2 결과에 적용)
- `discrepancy.py` — 보완적 역할 유지 (경험 불일치, 미래 날짜 등 LLM 불필요 체크)

### DiscrepancyReport 모델
- `alerts` JSONField에 integrity_flags 항목 추가
- `llm_assessment` 필드 활용 (현재 미사용)
- `scan_version`을 "v4"로 업데이트

### UI 표시
- `review_notice_items`에 integrity_flags 통합
- 기존 severity 체계(RED/YELLOW/BLUE) 유지
- `detail`, `reasoning` 필드를 상세 페이지에 표시

## 비용 분석

일반적 이력서 (경력 4-5개, 단일 언어):
- Step 1: ~3K input + ~2K output
- Step 1.5: ~2K input + ~0.5K output
- Step 2: 5개 그룹 × ~0.5K each = ~2.5K
- **총**: ~10K 토큰, Gemini Flash 기준 비용 미미

다국어 이력서 (국/영/일, 경력 4개):
- Step 1: ~5K input + ~4K output (섹션 수 비례)
- Step 1.5: ~4K input + ~0.5K output
- Step 2: 5개 그룹 × ~1K each = ~5K
- **총**: ~15K 토큰

병렬 처리로 Step 2의 wall-clock time은 가장 큰 그룹 1건의 처리 시간과 동일.

## 테스트 결과 요약

30건의 이력서에 대해 프롬프트를 v1~v11까지 반복 개선하며 테스트.
5건의 E2E 테스트(Drive 다운로드 → 텍스트 추출 → LLM 호출)에서 검증.

| 후보자 | 기대 | 최종 결과 |
|--------|------|----------|
| 권영화 | TP (국문/일문 7년 차이) | DATE_CONFLICT 탐지 ✓ |
| 오지영 | TP (2개월 vs 11개월) | DATE_CONFLICT 탐지 ✓ |
| 백세라 | TP (정규직 장기 중복) | PERIOD_OVERLAP 탐지 ✓ |
| 곽민선 | TN (변호사 겸업) | 겸업 YELLOW (합리적) |
| 고건창 | TN (오타 + 회사 내 이동) | 정상 처리 (오타 제외, 세부 통합) ✓ |

### 핵심 발견사항
- 추출과 정규화를 분리해야 정보 소실 없이 위조 탐지 가능
- LLM에게 판단을 맡길 때 맥락(WHY)이 없으면 severity가 불안정
- PERIOD_OVERLAP은 구조화 데이터로 규칙 계산이 LLM보다 정확
- "현재" 처리는 프롬프트보다 구조(전체 기간 우선 선택)로 해결해야 안정적
- 시멘틱 그룹핑으로 Step 2의 태스크를 단순화하면 판단 품질 향상

## 구현 범위
