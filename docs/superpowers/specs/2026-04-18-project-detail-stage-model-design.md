# 프로젝트 상세 페이지 단계 모델 설계 (Stage Model Redesign)

> **작성일**: 2026-04-18
> **상태**: 브레인스토밍 결과. 이후 writing-plans로 구현 계획 수립 예정.
> **범위**: 프로젝트 상세 페이지의 업무 프로세스 8단계 재정의. 단계를 프로젝트 레벨 / 후보자 레벨로 분류, 각 단계의 To-Do·판정 기준·자동화·건너뛰기 정책을 명확화.

---

## 1. 배경 — 왜 재설계하는가

### 기존 한계

현재 구현(Phase A + B 완료 시점)은 8단계를 **Application(후보자별) 진행 바**에 일렬로 표시하고 있음. 이 구조는 다음 문제를 야기한다.

1. **"서칭"이 후보자별 단계가 되는 논리적 모순** — 상세 페이지는 후보자 카드가 생성된 이후에야 의미가 있는데, 그 진입 시점에 "서칭" 단계가 있다는 것이 어색함.
2. **"고객사 제출"이 후보자별 단계가 되는 한계** — 헤드헌팅 실무에서는 여러 후보자를 묶어 배치로 제출하는 경우가 흔하지만, 후보자별 단계로만 모델링하면 이를 표현할 수 없음.
3. **방법 분기가 단계 진행과 혼동됨** — 전화/이메일/카톡 같은 채널 선택이 시스템 관점에서 의미가 없는데도 단계 구성에 영향을 주고 있었음.
4. **단계별 판정 기준 불명확** — 각 단계의 "성공/실패/완료/미완료" 기준과 "건너뛰기 허용 여부"가 일관되지 않음.

### 이 설계가 풀려는 문제

- 단계를 **프로젝트 레벨** / **후보자 레벨** 로 구분
- 각 단계에 **To-Do / 완료 / 실패/드롭 / 건너뛰기 / 자동화** 를 명시적으로 정의
- 방법(채널)은 **시스템 분기가 필요한 경우**(예: 이력서 수집의 DB/이메일/업로드)에만 UI 분기, 단순 채널 선택은 제외
- **"시스템은 다음에 뭘 할지 가이드하는 툴"** 이라는 철학을 구조에 반영

## 2. 페이지 구조

```
┌─────────────────────────────────────────────┐
│ [헤더] 제목 · JD · 상태 · 오너 · 컨설턴트    │
├─────────────────────────────────────────────┤
│ [A] 프로젝트 레벨 작업 영역                  │
│   · 서칭 도구 (4 채널)                       │
│   · 이력서 제출 관리 (배치 제출용)           │
├─────────────────────────────────────────────┤
│ [B] 후보자 카드 리스트                       │
│   각 카드 = 7단계 진행바 + 현 단계 To-Do     │
└─────────────────────────────────────────────┘
```

### 영역 A (프로젝트 레벨)

- **서칭 도구**: 후보자 발굴. 프로젝트 생성 직후부터 종료 시점까지 **상시 활성**
- **이력서 제출 관리**: 대기 중인 후보자 중 여러 명을 묶어 배치 제출하는 UI

### 영역 B (후보자 카드 리스트)

- 각 후보자 = 1 Application = 1 카드
- 카드마다 **7단계 진행바** (접촉 → 이력서 준비 → 사전 미팅 → 이력서 작성 → 이력서 제출 → 면접 → 입사)
- "서칭"은 진행바에 포함하지 않음 (프로젝트 레벨)
- 카드 생성 시점 = 서칭 단계 통과 완료 상태

## 3. 단계 레벨 분류

### 프로젝트 레벨 단계

| 단계 | 특성 |
|---|---|
| **서칭** | 상시 활성. 프로젝트 종료까지 언제든 재개 가능. drop 여부·배치 제출 여부와 무관하게 추가 서칭 가능. "완료"라는 개념 없음 — 활성/비활성만 존재. |

### 후보자 레벨 단계 (카드 진행바)

| 순서 | 단계 |
|---|---|
| 1 | 접촉 |
| 2 | 이력서 준비 *(기존 명: 이력서 수집)* |
| 3 | 사전 미팅 |
| 4 | 이력서 작성(제출용) *(기존 명: 제출 준비)* |
| 5 | 이력서 제출 *(기존 명: 고객사 제출)* |
| 6 | 면접 |
| 7 | 입사 |

### 하이브리드 — "이력서 제출" 단계

- 카드 진행바에는 항상 포함 (7단계 일관성)
- **실행 경로 2가지 공존**:
  - **배치**: 영역 A의 "제출 관리"에서 여러 명 묶어 제출 → 각 카드의 해당 단계가 자동으로 "제출 완료" 상태로 전환
  - **개별**: 각 카드 자체에서 "단독 제출" 버튼 → 해당 카드만 제출
- **프로젝트 레벨에 `submission_mode` 필드를 두지 않음.** 제출 이벤트마다 컨설턴트가 선택.
- 한 프로젝트 안에서 "2명은 배치, 1명은 개별" 혼합 허용.

## 4. 서칭 단계 상세

### 4.1 후보자 등록 과정

#### DB 서칭 (현 단계 구현 범위)

1. 프로젝트 상세 영역 A에서 **"DB 서칭"** 클릭
2. 기존 `candidates` 페이지 재사용 — JD 기반 필터 + 스코어링 결과 리스트
3. URL에 프로젝트 ID 전달 → "이 프로젝트용" 모드로 진입
4. 컨설턴트가 후보자 카드에서 **"프로젝트에 추가"** 버튼 (단일) 또는 체크박스 다중 선택 → "선택 N명 추가"
5. 선택된 후보자별 `Application` 레코드 생성
6. 각 Application은 "접촉" 단계부터 활성 (서칭은 통과 표시)

#### 외부 채널 (현 단계 UI만, 자동화는 추후)

1. 영역 A에 4개 채널 버튼: **잡코리아 · 사람인 · LinkedIn · 이메일**
2. 클릭 → 현재는 **"준비 중"** 상태 표시
3. *(추후)* AI가 JD 기반 채널별 포스팅 문구 생성 → 컨설턴트 편집 → 실행
4. *(추후)* 들어온 응답 수신 목록 화면
5. *(추후)* 컨설턴트가 응답 중 선별 → "프로젝트에 추가" → Application 생성 (자동 등록 ❌ / 수동 등록 ✓)

### 4.2 상시 활성 원칙

- drop과 무관하게 **언제든 추가 서칭 가능**
- 고객사의 "추가 후보 요청" 같은 외부 이벤트에 즉시 대응 가능해야 함
- "서칭 종료" 상태 없음 — 프로젝트 종료(입사 확정 또는 실패)까지 열려 있음
- drop 발생 시 자동 서칭 재활성화 트리거도 불필요 (상시 활성이므로)

### 4.3 판정 기준

| 상태 | 조건 |
|---|---|
| 활성 | 프로젝트 OPEN 상태 동안 항상 |
| 진행 | 컨설턴트가 서칭 작업을 수행 중 (상태 플래그 불필요) |
| 비활성 | 프로젝트 CLOSED 시 자동 |

### 4.4 권한

- 서칭 도구 사용: **오너 + 컨설턴트 공용**
- 외부 채널 계정 연동: **컨설턴트 개인 단위** (본인 LinkedIn/잡코리아/이메일 계정)

## 5. 후보자 카드 7단계 상세

### 공통 완료 판정 원칙 (Option C)

- 산출물이 시스템 데이터로 잡히는 단계 → **자동 판정** (이력서 준비·사전 미팅·이력서 작성(컨펌 포함)·면접·입사)
- 산출물이 없거나 외부 이벤트인 단계 → **컨설턴트 체크** (접촉·이력서 제출)
- 단, "이력서 작성(제출용)"은 자동 생성되더라도 **컨펌 필수**

### 5.1 접촉

| 항목 | 내용 |
|---|---|
| To-Do | JD 공유 + 응할 의사 확인 |
| 완료 | **컨설턴트 체크** (선택 기록: 응답 요지 — 긍정/부정/보류) |
| 실패/드롭 | 부정 응답 · 장기 무응답 → drop (`candidate_declined`) |
| 건너뛰기 | 허용 (이미 합의된 후보·오너가 직접 넘긴 경우) |
| 자동화 | 없음 |

### 5.2 이력서 준비

| 항목 | 내용 |
|---|---|
| To-Do | 이력서 파일 확보 |
| 완료 | **자동** — `Resume` 레코드가 Application에 연결됨 |
| 실패/드롭 | 제출 거부 → drop (`candidate_declined`) |
| 건너뛰기 | 허용 (이미 충분한 정보가 있는 드문 경우) |
| 자동화 | Phase B 완료 (DB 재사용 · 업로드) / 이메일 자동 수집은 미구현 |

### 5.3 사전 미팅

| 항목 | 내용 |
|---|---|
| To-Do | (a) 미팅 일정 조율 → (b) 미팅 진행 → (c) 결과 기록 |
| 완료 | **자동** — `MeetingRecord` 에 `held_at` + 결과(요약) 기록됨 |
| 미완료 상태 세분화 | ① 일정 미확정 → ② 일정 확정, 진행 전 → ③ 진행 완료, 기록 대기 |
| 실패/드롭 | 미팅에서 부적합 판단 → drop (`unfit`) |
| 건너뛰기 | 허용 (미팅 생략하고 바로 제출하는 경우) |
| 자동화 | 녹음 업로드 → STT → AI 요약 → 컨설턴트 편집 / 수기 대체 허용 (**별도 설계**) |

### 5.4 이력서 작성 (제출용)

| 항목 | 내용 |
|---|---|
| To-Do | 고객사 제출용 이력서 문서 작성 + **컨설턴트 컨펌** |
| 완료 | 문서 자동 생성 + **컨설턴트 컨펌 버튼** (자동 완료 판정 불가) |
| 실패/드롭 | 작성 과정에서 부적합 판단 → drop (`unfit`) |
| 건너뛰기 | **불가** (고객사 제출 전 필수) |
| 자동화 | 목표 = 자동 생성 / 현재 = UI만, 실제 생성 로직은 **별도 설계** (고객사별 템플릿·기본 템플릿 선택) |

### 5.5 이력서 제출

| 항목 | 내용 |
|---|---|
| To-Do | 고객사에 이력서 전달 |
| 완료 | **컨설턴트 완료 버튼** (자동 판정 없음) |
| 실패/드롭 | **없음** (보내면 끝. 드롭 시나리오 없음) |
| 건너뛰기 | **불가** |
| 자동화 | 불필요 (컨설턴트 수동 처리) |
| 실행 경로 | 배치(영역 A) 또는 개별(카드) 양쪽 허용 |

### 5.6 면접

| 항목 | 내용 |
|---|---|
| To-Do | (a) 면접 결과 대기 · (b) After Interview Review 기록(선택) |
| 완료 | **자동** — Review 등록 시 OR **컨설턴트 버튼** 으로 완료 |
| 실패/드롭 | 면접 탈락 → drop (`client_rejected`) |
| 건너뛰기 | 허용 (리뷰 생략 가능) |
| 자동화 | Review 녹음 → STT → AI 요약 (**5.3과 공통 파이프라인**) |

### 5.7 입사

| 항목 | 내용 |
|---|---|
| To-Do | 입사일 확정 |
| 완료 | **자동** — `Application.hired_at` 세팅 |
| 실패/드롭 | Offer 철회 · 후보자 거부 → drop (`candidate_declined`) |
| 건너뛰기 | 불가 (최종 단계) |
| 자동화 | 프로젝트 자동 종료 트리거 (구현됨) |

## 6. 프로젝트 레벨 판정 기준

| 판정 | 조건 |
|---|---|
| 성공 | 1명이라도 입사 확정 (`Application.hired_at` 세팅) |
| 실패 | 전원 drop + 서칭 종료 선언, 또는 기간 만료(deadline 경과) |
| 진행 중 | 위 조건에 해당하지 않는 OPEN 상태 |

## 7. 네이밍 변경 요약

| 기존 | 변경 |
|---|---|
| 이력서 수집 | **이력서 준비** |
| 제출 준비 | **이력서 작성(제출용)** |
| 고객사 제출 | **이력서 제출** |

영향 범위: `STAGES_ORDER`, `STAGE_FROM_ACTION_TYPE`, `STAGE_GATES`, 템플릿 파티얼, 라벨 텍스트 전반.

## 8. 별도 설계 필요 항목 (향후 구현 대상)

구현 계획(writing-plans) 시점에 각 항목을 별도 설계 문서로 분리하여 단계적으로 진행.

1. **이메일 자동 이력서 수집** (5.2 이력서 준비의 3번째 경로)
2. **고객사별 제출용 이력서 템플릿 + 기본 템플릿 선택 + 자동 생성** (5.4 핵심)
3. **녹음 → STT → AI 요약 파이프라인** (5.3 사전 미팅 + 5.6 면접 리뷰 공용)
4. **AI 포스팅 문구 생성** (4.1 외부 채널)
5. **외부 계정 연동 프레임워크** (잡코리아/사람인/LinkedIn 등 OAuth 또는 크레덴셜)
6. **채널별 자동 게시 + 응답 취합** (4.1 외부 채널 실행)
7. **JD 보이스 입력 에이전트** (프로젝트 생성 플로우 — 본 설계 범위 밖이지만 필수 전제)
8. **JD 필수 항목 스키마 정의** (위와 동일)

## 9. 현재 구현 상태 vs 이 설계

| 영역 | Phase A+B 완료 상태 | 이 설계에서의 요구 | 갭 |
|---|---|---|---|
| 8단계 상수 | 구현됨 | 7단계 + 프로젝트 레벨 서칭 | 재구조화 |
| 네이밍 | sourcing/resume/prep_submission/client_submit | 서칭/이력서 준비/이력서 작성/이력서 제출 | 라벨 변경 |
| 서칭 영역 | 없음 | 프로젝트 레벨 A 영역 신규 | 신규 UI |
| 배치 제출 관리 | 없음 | 프로젝트 레벨 A 영역에 추가 | 신규 UI + 로직 |
| 이력서 준비 3방법 | 구현됨 (Phase B) | 그대로 유지 | — |
| 사전 미팅 세부 단계 | 없음 | 일정/진행/기록 3단계 | 세분화 |
| 이력서 작성 컨펌 | 없음 | 컨설턴트 컨펌 필수 | 신규 UX |
| After Interview Review | 없음 | 녹음/수기 선택 등록 | 신규 |

## 10. 다음 단계

이 설계 문서를 바탕으로 **writing-plans 스킬**로 Phase C 구현 계획을 작성한다. 계획은 다음 단위로 분리:

1. **모델·상수 재구조화** (STAGES_ORDER 재정의, 네이밍 변경, 마이그레이션)
2. **프로젝트 상세 페이지 레이아웃 분리** (영역 A + B)
3. **영역 A 서칭 도구 — DB 서칭 연결**
4. **영역 A 서칭 도구 — 외부 채널 UI 플레이스홀더**
5. **영역 A 이력서 제출 관리 (배치)**
6. **후보자 카드 재구성 (7단계, 단계별 To-Do 파티얼)**
7. **각 단계별 세부 UX** (5.3 사전 미팅 세분화, 5.4 컨펌 UI, 5.6 면접 리뷰 등)
8. **별도 설계 문서로 분리된 8개 항목** — 각각 독립 계획 수립

각 단위는 독립된 구현 스텝으로 분리 가능하며, 1·2·6번이 선행되어야 나머지를 얹을 수 있다.
