## 상세 점검 결과

### 1. 플로팅 버튼이 음성 검색임을 즉시 인지할 수 있는가? ✅ 통과

**코드 근거:**
- `search.html:105-108` — FAB에 마이크 SVG 아이콘 사용. 일반적인 챗봇 아이콘(말풍선)이 아니라 마이크 아이콘이므로 "음성"이라는 기능을 시각적으로 전달함.
- `aria-label="음성 검색"` (line 104) — 접근성 라벨 적절.
- `w-14 h-14 lg:w-[60px] lg:h-[60px]` — 터치 타겟 충분 (56px/60px).
- `bg-primary` (#4A56A8) — 페이지 배경(gray-50)과 충분한 대비.

**개선 권장 (Minor):**
- 마이크 아이콘만으로는 "음성 녹음"인지 "음성 검색"인지 구분이 모호할 수 있음. 아이콘 하단에 작은 텍스트 라벨("검색") 또는 돋보기 배지 추가를 고려할 수 있으나, 현재 온보딩 툴팁("음성으로 검색해보세요")이 보완 역할을 하므로 필수는 아님.

**코드 근거:**
- `chatbot_modal.html:58-84` — voice-input-area가 기본 표시. 중앙에 큰 마이크 버튼(`w-14 h-14`) 배치.
- `chatbot_modal.html:87` — text-input-area는 `class="hidden"`으로 기본 숨김.
- `chatbot_modal.html:78` — 상태 텍스트 "마이크를 눌러 말씀하세요"가 명확한 CTA.
- `chatbot_modal.html:81-83` — 텍스트 입력 전환 링크는 `text-xs text-gray-400`로 작고 보조적인 위치에 배치.

**판정:** 보이스 퍼스트 설계 원칙에 부합. 시각적 계층이 명확함 (큰 마이크 > 안내 텍스트 > 작은 키보드 링크).

### 3. 녹음 중 충분한 시각적 피드백이 있는가? ⚠️ 부분 통과

**통과 요소:**
- `chatbot_modal.html:61-62` — 녹음 중 펄스 링 2개 (animate-ping + animate-pulse) 표시. 시각적으로 "활성 상태"임을 전달.
- `chatbot.js:361-362` — 버튼 색상 bg-primary → bg-red-500 전환 + scale-110 확대. 상태 변화 명확.
- `chatbot.js:71-72` — 마이크 아이콘 → 정지(사각형) 아이콘 전환. "녹음 중지" 행동 유도 적절.
- `chatbot.js:102-121` — 녹음 타이머 표시 (MM:SS). 남은 시간 10초 이내 시 animate-pulse 추가.
- `chatbot.js:365` — 상태 텍스트 "듣고 있습니다..." 표시.

**문제 (Major):**
- **실시간 오디오 파형/레벨 미터 부재.** 현재 피드백은 정적 애니메이션(ping/pulse)과 타이머뿐임. 실제로 마이크가 소리를 감지하고 있는지 사용자가 확인할 방법이 없음. 예를 들어, 마이크가 잘못 선택되었거나 음소거 상태일 때도 동일한 애니메이션이 표시됨. `AudioContext.analyser`를 사용한 간단한 볼륨 레벨 표시가 필요함.

**개선 권장:**
- (Major) Web Audio API의 `AnalyserNode`를 활용하여 실시간 입력 볼륨을 시각적으로 표시하면, 마이크가 실제로 소리를 수신하고 있다는 확신을 줄 수 있음. 원형 버튼 주변 링의 크기나 투명도를 볼륨에 연동하는 방식이 효과적.
- (Minor) 녹음 타이머가 마이크 버튼 아래에 별도 요소로 존재하나(`mic-timer`), 상태 텍스트(`mic-status-text`)와 세로로 나열되어 시선 분산. 타이머를 상태 텍스트에 인라인 통합 ("듣고 있습니다... 00:15") 하면 정보 밀도 개선.

### 4. 텍스트 입력과 음성 입력의 관계가 적절한가? ✅ 통과

**코드 근거:**
- `chatbot.js:38-53` — showTextInput/hideTextInput이 두 영역을 상호 배타적으로 전환. 동시 표시 없음.
- `chatbot_modal.html:89` — 텍스트 모드에서 마이크 아이콘 버튼으로 음성 모드 복귀 가능.
- `chatbot_modal.html:81-83` — "키보드로 입력" 토글은 `text-xs text-gray-400`으로 보조 수준. 음성이 주 입력이라는 계층 유지.

**판정:** 음성=주, 텍스트=보조 관계가 UI 계층에서 일관되게 유지됨.

### 5. 음성→결과 표시 흐름이 끊기지 않는가? ✅ 통과

**코드 근거:**
- 전체 흐름: 녹음 종료 → `handleRecordingComplete()` → setMicState("processing") + "음성 인식 중..." 표시 → STT 결과 수신 → `appendUserMessage(text, true)` (🎤 태그) → `appendThinking()` ("생각 중...") → `doSearch()` → 결과 수신 → `removeThinking()` + AI 메시지 표시 + `updateStatusBar()` + `refreshCandidateList()`.
- `chatbot.js:296-299` — 음성 입력 시 🎤 이모지로 음성 발화였음을 표시.
- `chatbot.js:270-272` — 검색 완료 후 URL에 session_id 반영 + 후보자 리스트 HTMX 새로고침.
- `chatbot.js:421-429` — 검색 상태바에 쿼리와 결과 수 표시, 클릭하면 모달 재진입.

**개선 권장 (Minor):**
- `setMicState("processing")` 단계와 `appendThinking()` 단계가 별도임. 사용자는 "음성 인식 중..." → (결과) → "생각 중..." 두 단계를 거치는데, 이 전환이 매끄러운지는 네트워크 속도에 따라 다름. STT가 빠르면 "음성 인식 중..."이 순간적으로 번쩍이고 바로 "생각 중..."으로 넘어가 시각적 잡음 발생 가능. 두 단계를 하나의 연속 피드백("음성을 분석하고 있습니다...")으로 통합하는 것도 고려할 만함.

### 6. 에러 상태 안내가 구체적인가? ⚠️ 부분 통과

**통과 요소:**
- `chatbot.js:381-393` — **마이크 권한 거부:** 빨간 배경 카드 + "브라우저 주소창 왼쪽 🔒 아이콘 → 마이크 → 허용으로 변경해주세요" 구체적 안내 + 텍스트 대안 링크 제공. 우수함.
- `chatbot.js:210-215` — **무음 감지:** "음성이 감지되지 않았습니다. 다시 말씀해주세요." 명확.
- `chatbot.js:175-179` — **녹음 과대:** blob 10MB 초과 시 "녹음이 너무 깁니다. 60초 이내로 다시 시도해주세요."
- `chatbot.js:221-231` — **네트워크/인증/속도 제한/서버 오류:** 각각 분기 처리됨.
  - 오프라인: "인터넷 연결을 확인해주세요."
  - 인증: "로그인이 필요합니다. 페이지를 새로고침해주세요."
  - 429: 서버 메시지 표시 또는 기본 "요청이 너무 많습니다..."
  - 기타: "음성 인식에 실패했습니다. 텍스트로 입력해주세요."
- `chatbot.js:274-286` — 검색 API 에러도 동일한 분류 처리.

**문제 (Major):**
- **마이크 권한 거부 메시지에서 OS/브라우저별 차이 미고려.** `showMicPermissionError()`는 "브라우저 주소창 왼쪽 🔒 아이콘"이라고 안내하지만, Safari(iOS)에서는 주소창에 🔒 아이콘이 없고 설정 앱에서 권한을 변경해야 함. 또한 Firefox는 권한 관리 UI가 다름. 모바일 웹이 주 타겟이라면 iOS Safari에 대한 안내가 특히 중요.

**개선 권장 (Minor):**
- `startRecording()`의 `.catch()` (`chatbot.js:159-161`)가 단일 `showMicPermissionError()`만 호출하지만, `getUserMedia` 거부는 `NotAllowedError` 외에도 `NotFoundError`(마이크 하드웨어 없음), `NotReadableError`(다른 앱이 마이크 점유) 등 여러 원인이 있음. 에러 종류에 따른 안내 분기가 없어 "권한 거부" 메시지가 잘못 표시될 수 있음.

### 7. 모바일에서 사용하기 편한가? ⚠️ 부분 통과

**통과 요소:**
- `chatbot_modal.html:9` — 모바일에서 `bottom-0 left-0 right-0 h-[85vh] rounded-t-2xl`로 바텀시트 패턴. 적절.
- `chatbot_modal.html:27-29` — 모바일 드래그 핸들 표시 (`lg:hidden`). 바텀시트 관례에 부합.
- `input.css:25-27` — `safe-area-pb` 정의 (`padding-bottom: env(safe-area-inset-bottom)`). iPhone 노치/홈바 대응.
- `chatbot_modal.html:56` — 입력바에 `safe-area-pb` 클래스 적용됨.
- `search.html:7-9` — 검색 페이지에서 사이드바/하단 네비를 제거하여 전체 너비 활용.

**문제 (Major):**
- **드래그 핸들이 시각적 장식일 뿐 실제 드래그 동작 미구현.** `chatbot_modal.html:27-29`에 드래그 핸들 UI가 있지만, 터치 이벤트 핸들러가 없음 (`chatbot.js`에 touchstart/touchmove/touchend 로직 부재). 바텀시트에 드래그 핸들이 있으면 사용자는 아래로 스와이프하여 닫을 수 있다고 기대함. 현재는 닫기 버튼(최소화 아이콘) 또는 오버레이 탭만 가능.

**개선 권장 (Minor):**
- `chatbot_modal.html:19-23` — 닫기 버튼의 아이콘이 가로선(─) 하나(`M20 12H4`)로 "최소화" 의미를 전달하려는 것으로 보이나, 일반적인 닫기(X) 아이콘과 다르고, 모바일에서 터치 영역(`p-2.5`)이 충분하긴 하지만 44px 기준에 약간 부족할 수 있음. `p-3`으로 여유를 확보하는 것이 안전.

### 8. 첫 사용자가 사용법을 바로 이해할 수 있는가? ✅ 통과

**코드 근거:**
- `chatbot.js:462-480` — 첫 방문 시 FAB 위에 "음성으로 검색해보세요" 툴팁 표시. 6초 후 자동 숨김. `localStorage`에 `synco_onboarding_seen` 플래그로 1회만 표시. 모달 열기 시에도 즉시 숨김 처리.
- `search.html:93-98` — 툴팁 UI: 검은 배경 말풍선 + 꼬리 화살표 + animate-bounce (3회 반복). 시각적 주목성 확보.
- `chatbot_modal.html:34-51` — 모달 내 환영 메시지에 "마이크를 눌러 음성으로 검색하세요" + 예시 칩 3개 제공. 첫 사용자가 즉시 시도 가능.

**개선 권장 (Minor):**
- 온보딩 툴팁이 6초이고 bounce 3회(animation-iteration-count:3, duration:2s)로 6초 동안 움직임. 타이밍 자체는 적절하나, 사용자가 다른 곳을 보고 있을 때 놓칠 수 있음. 첫 방문이지만 툴팁을 놓쳤을 경우, 모달 내 환영 메시지가 보완 역할을 하므로 큰 문제는 아님.

## 수정 필요 사항 (우선순위별)

### Major

1. **실시간 오디오 레벨 피드백 부재** (점검 항목 3)
   - **현상:** 녹음 중 정적 CSS 애니메이션(ping/pulse)만 표시. 실제 마이크 입력 유무를 사용자가 확인할 수 없음.
   - **위치:** `chatbot.js` — `startRecording()` 함수 (line 136-162)
   - **수정 방향:** `AudioContext` + `AnalyserNode`로 실시간 볼륨을 감지하여, 펄스 링의 크기/투명도 또는 마이크 버튼 주변 시각 요소에 반영. MediaRecorder와 병행 가능.

2. **바텀시트 드래그 제스처 미구현** (점검 항목 7)
   - **현상:** 모바일 바텀시트에 드래그 핸들 UI가 있으나 실제 터치 드래그 동작 없음.
   - **위치:** `chatbot_modal.html` line 27-29 (핸들 UI), `chatbot.js` (이벤트 핸들러 부재)
   - **수정 방향:** touchstart/touchmove/touchend 이벤트로 바텀시트 스와이프 닫기 구현. 또는 드래그 핸들 UI를 제거하여 사용자 기대 불일치 방지.

3. **마이크 에러 분류 미흡** (점검 항목 6)
   - **현상:** `getUserMedia` 실패 시 모든 에러를 "권한 거부"로 처리. `NotFoundError`, `NotReadableError` 등 구분 없음. iOS Safari에 대한 별도 안내 없음.
   - **위치:** `chatbot.js` line 159-161 — `.catch(function () { showMicPermissionError(); })`
   - **수정 방향:** catch에서 에러 객체의 `name` 속성을 확인하여 분기. `NotFoundError` → "마이크를 찾을 수 없습니다", `NotReadableError` → "마이크가 다른 앱에서 사용 중입니다" 등. iOS 감지 시 설정 앱 안내 추가.

### Minor

1. **STT→검색 전환 시 이중 로딩 상태** (점검 항목 5)
   - **현상:** "음성 인식 중..." → "생각 중..." 두 단계가 빠르게 전환될 때 시각적 잡음.
   - **수정 방향:** 두 상태를 하나의 연속 진행 표시로 통합 검토.

2. **녹음 타이머와 상태 텍스트 분리 배치** (점검 항목 3)
   - **현상:** `mic-status-text`와 `mic-timer`가 수직 나열. 정보 분산.
   - **수정 방향:** 타이머를 상태 텍스트에 인라인 통합.

3. **모달 닫기 버튼 터치 영역** (점검 항목 7)
   - **현상:** `p-2.5`는 약 40px. 모바일 최소 44px 권장에 미달 가능.
   - **수정 방향:** `p-3`으로 확대.

4. **FAB 아이콘 라벨 보강** (점검 항목 1)
   - **현상:** 마이크 아이콘만으로 "검색" 기능 전달이 간접적.
   - **수정 방향:** 선택적으로 아이콘 하단 작은 텍스트 또는 돋보기 배지 추가.

5. **`prefers-reduced-motion` 시 녹음 피드백 대안** (점검 항목 3)
   - **현상:** `input.css:16-22`에서 reduced-motion 시 모든 애니메이션 비활성화. 녹음 중 펄스링, 타이머 pulse 등이 사라지면 녹음 상태를 인지하기 어려움.
   - **수정 방향:** reduced-motion 환경에서는 색상 변화 또는 정적 아이콘 교체 등 비애니메이션 방식의 녹음 피드백 보장.

6. **온보딩 놓침 시 재진입점 부재** (점검 항목 8)
   - **현상:** 온보딩 툴팁은 1회만 표시. 놓친 사용자를 위한 별도 안내 없음.
   - **수정 방향:** 모달 내 환영 메시지가 보완 역할을 하고 있어 큰 문제는 아니나, 검색 페이지 상단 빈 상태에 "음성 검색으로 시작해보세요" 같은 힌트 추가 검토.

## 전반적 평가

보이스 퍼스트 인터페이스로서의 기본 설계가 잘 되어 있음. 음성=주, 텍스트=보조의 UI 계층이 일관되고, 모달 내 초기 메시지와 예시 칩으로 첫 사용자의 진입 장벽을 낮추고 있음. 에러 처리도 대부분의 시나리오를 커버하고 있음.

가장 큰 개선 포인트는 **녹음 중 실시간 오디오 피드백**(사용자가 "마이크가 작동하고 있나?"를 확인할 수 없음)과 **바텀시트 드래그 제스처**(UI와 동작의 불일치). 이 두 가지는 보이스 퍼스트 인터페이스의 신뢰감과 모바일 사용성에 직접적으로 영향을 미치므로 우선 개선이 권장됨.
