# 01. 엑셀 분석 — 현실의 헤드헌팅 업무 구조

기준 자료는 Google Drive 공유문서함의 `Search Status.xlsx`. 이 문서 하나가 synco가 최종적으로 대체하려는 현재의 업무 트래킹 시스템이다.

## 1. 파일 개요

- **파일명**: Search Status.xlsx
- **Drive file_id**: `1hOTakXTuCj126JCuKAxqcEKToemtFcpJ`
- **형식**: Drive에 업로드된 xlsx (native Google Sheets 아님). gid 기반 탭 식별 불가
- **크기**: 1.5MB, ~6,500행, 31개 컬럼 (탭별 상이)

## 2. 컬럼 스키마 — 명목 vs 실제 사용

헤더 이름과 실제 저장 내용이 컨설턴트마다 다르다. 이게 시스템화의 첫 번째 난관이다.

| 컬럼 (헤더) | 명목상 의미 | 실제 관찰 |
|---|---|---|
| **Client** | 고객사 | 대부분 회사명. **전병권은 `Veolia (종료)`, `오에스피 (종료)` 같이 접미사로 프로젝트 상태 표기** |
| **Position** | 포지션 제목 | 대부분 정상. **임성민은 `Cloud Infra Engineer (Kubernetes/AWS)    OPEN` 같이 접미사로 상태 표기** |
| **Position/Title** | — | 실제로는 **후보자 경력 요건** (`5년↑`, `대리~과장`, `10~15년, 80~86년생`) |
| **Candidate** | 후보자 이름 | 그대로 |
| **Source** | 발견 출처 | `LinkedIn`, `P&J`, `사람인`, `Job Korea`, `인크루트`, `DB`, `PNJ` 등 |
| **Contact Method** | 접촉 채널 | 박정일/박지영: `Message`, `Email` (채널)<br>김현정/정호열: `Searching`, `Job Posting` (발견 방식)<br>**전병권: `서칭-제안메일-회신대기중-진행확정` (파이프라인 로그!)** |
| **Contact** | 최초 접촉일 | 날짜 |
| **Submission** | 서류 제출일 | 박정일/김현정: 날짜<br>**전병권/임성민: `to PM (3/24)`, `to Client (3/3)` (핸드오프 단계)** |
| **1st / 2nd / 3rd Interview** | 면접 일자 | 날짜. 박정일은 3차에 `추가 제안` 같은 주석도 씀 |
| **Summary / Remark / 경력** | 요약 메모 | 주로 **경력 년수**, 자격증, 특이사항 태그 |
| **Result** | 결과 | **사실상 후보자 여정 로그**. 화살표 노테이션으로 전체 파이프라인 기록 |
| **Candidate's Information** | 후보자 정보 | `이름.출생년도.전직장.학력` 한 줄 요약 |

**핵심 발견**: Submission/Interview 컬럼은 생각보다 잘 안 쓰인다.
- 박지영 탭: 281행 전체에서 Submission/Interview 컬럼 **전부 비어있음**
- 정호열 탭: 1,457행 중 Submission 기입 2건, Interview 1건 이하
- 모든 Funnel 정보가 **`Result` 컬럼 자유 텍스트 안에 함축**되어 있음

## 3. 컨설턴트별 기록 스타일 — 6개의 서로 다른 체계

같은 시트를 공유하지만 **현실은 6개의 다른 시스템을 한 파일에 모아놓은 것**이다.

### 박정일 — 화살표 진행 로그 (가장 세밀)
```
제안 > JD전달 > 자유양식 이력서 접수 > 내부양식 이력서 접수 > 서류접수完 > 서류 합격 > 최종 합격 > 입사 포기 (개인사정)
```
Unique Result 값 210개. 입사 여부까지 추적.

### 박지영 — 축약 화살표
```
제안 > JD전달>지원완료>탈락
```
Interview 컬럼 전혀 안 씀. Result에 모든 것.

### 김현정 — 태그 체계 (가장 일관적)
```
부적합 (경력초과)
제안거절 (이직의사無)
지원철회 (심경변화)
1차면접 탈락
```
드롭 사유를 카테고리 태그로 정리. **재사용 가능한 분류 체계의 힌트를 주는 가장 중요한 샘플.**

Top 10 카테고리 (빈도순):
```
[1271] 제안
[ 117] 제안거절 (지원의사無)
[  81] 제안거절 (이직의사無)
[  80] 부적합 (학력조건)
[  79] 부적합 (경력초과)
[  73] 서류전형 탈락
[  66] 부적합 (직무불일치)
[  58] 부적합 (산업분야)
[  38] 제안거절 (직무불일치)
[  32] 부적합 (잦은이직)
```

### 정호열 — `지원/제안 => JD전달 => 이력서`
서식 불일치 (`=>`, `==>`, ` => `). 단계가 얕음 (대부분 이력서 제출까지).

### 전병권 — 파이프라인 + 핸드오프 (유일하게 명시적 프로젝트 상태 표기)
```
Contact Method: 서칭-제안메일-회신대기중-진행확정
Submission:     to Client (3/3)
Result:         1차 합격, 2차 준비중
```
**유일하게** `진행중`, `진행취소`, `대기중`, `1차 합격` 같은 프로젝트 레벨 워딩을 사용. 

### 임성민 — 단순 드롭 사유
```
경력미달 / 나이많음 / 외국인제외 / 본인포기(맥스 강남 통탄 무리)
```
샘플 적음 (22행).

## 5. 관찰된 프로젝트 라이프사이클 단계 (현실)

데이터에서 역추적한 실제 단계:

```
0. [포지션 수주]            ← 시트 밖 (사장이 클라이언트 대면)
    ↓
1. 서칭/접촉                "제안" / "서칭-제안메일" / Searching / Job Posting
    ↓
2. JD 공유/관심 확인        "JD전달" / "회사명 오픈"
    ↓
3. 이력서 수집              "자유양식 이력서 접수" / "내부양식 이력서 접수"
    ↓
4. 내부 PM 검토             "to PM (3/24)"
    ↓
5. 클라이언트 제출          "to Client (3/3)" / "서류접수完"
    ↓
6. 서류 심사                → 서류합격 / 서류탈락
    ↓
7. 면접 진행 (1~3차, 최종)  1차/2차/3차 컬럼 또는 Result 내 "1차면접 탈락"
    ↓
8. 최종 합격                "최종 합격"
    ↓
9. [오퍼 협상]              거의 기록 없음 ⚠️
    ↓
10. 입사 확정/입사          "12월1일 입사" / "입사 포기 (개인사정)"
```

종료 경로:
- ✅ **SUCCESS**: 10번 도달
- ❌ **FAIL**: 모든 후보자 드롭 또는 타임아웃
- 🚫 **CANCEL**: "진행취소", "탈락-회사진행 마감"
- ⏸️ **HOLD**: "대기중" / "대기" (드물게)

## 6. 컨설턴트별 규모·성과 측정치

| 컨설턴트 | 총 후보자 접촉 | 프로젝트 수 | 서브미션 | 1차면접 | 2차면접 | 3차면접 | 명시적 성공 |
|---|---|---|---|---|---|---|---|
| 박정일 | 1,955 | 89 | 48 | 11 | 13 | 26 | 8 |
| 박지영 | 281 | 35 | 0 | 0 | 0 | 0 | 3 |
| 김현정 | 2,200 | 80 | 120 | 32 | 11 | 0 | 5 |
| 정호열 | 1,457 | 94 | 2 | 1 | 1 | 1 | 1 |
| 전병권 | 86 | 31 | 23 | 7 | 1 | 0 | 2 |
| 임성민 | 11 | 2 | 3 | 0 | 0 | 0 | 0 |
| **합계** | **~6,000** | **313** | **~196** | **51** | **26** | **27** | **19** |

관찰:
- **박정일**이 구조적 컬럼 사용이 가장 활발 (Interview 컬럼 실사용)
- **김현정**은 서브미션까지 활발하지만 Interview 컬럼은 거의 안 씀 (Result 텍스트로 대체)
- **정호열**은 1,457 접촉 중 99%가 "서칭중"으로 보이지만, 사실 상당수가 과거에 클로즈된 프로젝트인데 아무도 closed 표시 안 한 stale 상태
- **임성민**은 거의 신규 데이터 없음

## 7. 주요 데이터 품질 이슈

### 7-1. Stale 프로젝트 — "아무도 닫지 않은 종료 프로젝트"
정호열 탭의 99개 프로젝트 중 97개가 휴리스틱상 "서칭중"으로 분류되지만, 최소 수개월~수년간 활동 없음. **마지막 활동 기반 auto-close 규칙이 필수**.

### 7-2. 프로젝트 레벨 상태 표기 부재
313개 프로젝트 중 명시적 상태 표기는 전병권 2건 + 임성민 1건 = 3건 (1%). **현재 시트 사용자들은 프로젝트 레벨 상태를 거의 관리하지 않는다.** synco가 강제해야 할 습관.

### 7-3. 병렬 단계의 일상화
단일 프로젝트 안에서 후보자들이 **동시에 여러 단계**에 분포. 예: `전병권 - Veolia - Sales Engineer (n=18)` 프로젝트는 한 시점에 `to PM 대기`, `to Client 대기`, `1차 합격·2차 준비중`, `본인포기`, `탈락` 등 6개 상태가 공존. **Project에 단일 enum을 부여하는 모델로는 표현 불가능**.

## 8. 이 분석이 재설계에 주는 교훈

1. **프로젝트 상태는 단일 값이 아니다.** 후보자별 상태의 집합에서 derive되어야 한다.
2. **드롭 사유 분류 체계는 이미 존재한다.** 김현정의 태그 시스템이 설계 기준이 된다.
3. **오퍼 협상 단계는 버려도 된다.** 6건/10,000건의 데이터가 말해준다.
4. **사전 미팅 단계는 시트에 없다.** 컨설턴트 핵심 업무임에도 기록 부재 → synco가 새로 만들어줄 기회.
5. **stale 처리 정책이 필수.** 자동 close 규칙 없이는 상태가 금세 noise가 된다.
