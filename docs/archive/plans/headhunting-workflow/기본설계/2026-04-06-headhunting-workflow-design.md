## 1. 개요

### 배경

엑스다임 헤드헌팅 업무의 전체 프로세스를 synco 플랫폼에서 관리할 수 있도록 시스템을 확장한다.

**현재 synco:** 후보자 중심 (이력서 추출 → 검수 → 검색)
**확장 후 synco:** 프로젝트 중심 (의뢰 접수 → 서칭 → 컨택 → 추천 → 면접 → 합격)

### 헤드헌팅 업무 프로세스

```
고객사 영업·계약
  → 인재채용 의뢰 접수 (JD 제공)
    → 공지 작성·포스팅
      → 서칭 (DB, 잡포털, LinkedIn, 지원자 접수, 지인 추천)
        → 후보자 검증 (전화, 화상, 사전면접)
          → 제출 이력서 작성 (고객사 양식에 맞게)
            → 고객사 면접 (2~3차)
              → 합격 시 조율 (연봉, 직책, 출근일 등)
```

### 핵심 설계 원칙

| 역할 | 엔티티 | 시스템에서의 위치 |
|------|--------|-----------------|
| **주인공** | Project (의뢰) | 모든 업무 흐름의 중심. 상태가 바뀌면서 진행 |
| **재료** | Candidate (후보자) | 프로젝트에 투입되는 인력 풀. 여러 프로젝트에 재사용 |
| **소비자** | Client (고객사) | 의뢰를 내고 결과물(추천 후보)을 받는 쪽 |
| **운영자** | User/Consultant | 재료를 가공해서 소비자에게 전달하는 실행 주체 |

### 사용자

- **헤드헌터(컨설턴트) 전용** — 고객사 담당자는 시스템에 접근하지 않음
- 고객사 정보는 헤드헌터가 직접 등록·관리

## 2. Django 앱 구조

**2-앱 구조** 채택:

```
clients/          → 고객사, 계약, 레퍼런스 데이터(대학·기업·자격증)
projects/         → 프로젝트 라이프사이클, 공지, 컨택, 추천, 면접, 오퍼
candidates/       → (기존) 후보자 DB, 이력서 추출, 검색
data_extraction/  → (기존) AI 추출 파이프라인
```

**선택 이유:** 프로젝트 흐름(컨택→추천→면접→오퍼)이 한 앱 안에서 완결되고, 앱 간 의존 최소화.

## 4. 메뉴 구조

프로젝트 중심으로 사이드바 재편:

```
┌─────────────────────────────┐
│  synco                      │
├─────────────────────────────┤
│  📊  대시보드                │  ← 메인 홈. 내 진행중 프로젝트 현황
│                             │
│  📋  프로젝트               │  ← 의뢰 목록 (멀티뷰)
│                             │
│  🏢  고객사                 │  ← 고객사 목록/등록
│                             │
│  👤  후보자 DB              │  ← 기존 candidates (검색/검수)
│                             │
│  📰  뉴스피드              │  ← 업계 뉴스 (부가기능)
│                             │
├─────────────────────────────┤
│  ⚙️  레퍼런스 관리          │  ← 대학 랭킹, 기업 분류, 자격증 DB
│  👤  설정                   │
└─────────────────────────────┘
```

- **진입점 = 대시보드** — 로그인하면 내 진행 중 프로젝트 현황 즉시 확인
- **프로젝트 = 1순위 메뉴** — 업무의 중심
- **후보자 DB = 서칭 도구** — 프로젝트 내에서도, 독립적으로도 접근 가능
- **레퍼런스 = 하단 설정 영역** — 자주 변경하지 않는 마스터 데이터

## 5. JD 분석 파이프라인

프로젝트 등록의 핵심. JD 파일을 입력받아 AI가 요구조건을 자동 추출하고, 서칭 필터로 변환한다.

### JD 입력 소스 (3가지)

| 소스 | 설명 |
|------|------|
| **파일 업로드** | PDF, Word, HWP 파일 직접 업로드 |
| **Google Drive** | 공유 폴더에서 JD 파일 선택 (기존 Drive 연동 활용) |
| **텍스트 입력** | JD 내용 직접 붙여넣기 |

### 추출 → 서칭 필터 자동 매핑

```
JD 업로드/선택
  │
  ├─ 텍스트 추출 (PDF/Word/HWP → raw text)
  │
  ├─ AI 분석 (Gemini structured output)
  │   → requirements JSON 생성
  │   → jd_analysis에 전체 분석 결과 저장
  │
  ├─ 서칭 필터 자동 세팅
  │   birth_year_from/to → 후보자 birth_year 필터
  │   gender → 성별 필터
  │   min/max_experience → 경력 필터
  │   certifications → PreferredCert 매칭
  │   education.fields → 전공 키워드 필터
  │   keywords → 스킬/경력 키워드 필터
  │   industry → 업종 경력 필터
  │
  └─ 후보자 자동 매칭 (적합도 점수)
      각 조건별 가중치 부여 → 종합 적합도 산출
      "높음/보통/낮음" 3단계
```

## 6. 화면 설계

### 5.1 프로젝트 멀티뷰

노션 스타일로 같은 데이터를 **4가지 관점**으로 전환하여 볼 수 있다.
모든 뷰는 같은 필터를 공유하며, 뷰만 전��하면 된다.

```
┌─ 프로젝트 ──────────────────────── [+ 새 의뢰] ─┐
│                                                 │
│  [▣ 보드]  [≡ 리스트]  [⊞ 테이블]  [📅 캘린더]   │
│                                                 │
│  필터: [내 담당▾] [고객사▾] [상태▾] [기간▾]       │
│  정렬: [경과일순▾]                               │
└─────────────────────────────────────────────────┘
```

| 뷰 | 용도 | 표시 방식 |
|-----|------|----------|
| **▣ 보드** (기본) | 파이프라인 전체 흐름 파악 | 라이프사이클 칼럼 칸반. 병목 한눈에 |
| **≡ 리스트** | 오늘 할 일 기준 파악 | 긴급도·다음 액션 기준 정렬 |
| **⊞ 테이블** | 전체 프로젝트 비교·분석 | 스프레드시트형. 컬럼 정렬/필터 |
| **📅 캘린더** | 면접·재컨택 일정 확인 | 일정 중심 월간/주간 뷰 |

## 7. 충돌 방지 시스템

충돌 방지는 **2개 레이어**로 동작한다:

| 레이어 | 시점 | 문제 |
|--------|------|------|
| **프로젝트 등록 시** | 프리랜서가 의뢰 받고 등록할 때 | 같은 고객사의 같은/유사 포지션이 이미 진행 중인지 모름 |
| **후보자 컨택 시** | 서칭 후 후보자에게 연락할 때 | 같은 후보자에게 중복 연락 |

### 6.1 프로젝트 등록 충돌 감지

**배경:** 프리랜서 컨설턴트는 개인적으로 클라이언트와 접촉하므로, 다른 컨설턴트가 같은 고객사로부터 이미 유사한 의뢰를 받았는지 알 수 없다.

**가시성 정책:** 충돌 시에만 공개 — 평소에는 자기 프로젝트만 보이고, 충돌 감지 시 "유사 프로젝트 존재 (담당: OOO)" 수준만 표시.

**매칭 기준:**

| 조건 | 판정 | 액션 |
|------|------|------|
| 같은 고객사 + 포지션명 거의 동일 | 🔴 높은 중복 가능성 | 등록 전 회사 승인 필수 |
| 같은 고객사 + 포지션 다름 | 🟡 참고 정보 | 회사 승인 필수 (다른 포지션 확인) |
| 다른 고객사 | 감지 없음 | 바로 등록 |

**포지션 유사도 판정:** 키워드 매칭 ("품질기획팀장" vs "품질기획파트장" → 핵심 키워드 일치) + JD 내용 비교.

#### 승인 플로우

```
프리랜서가 프로젝트 등록
  │
  ├─ 충돌 없음 → 바로 등록 완료
  │
  └─ 충돌 감지 → 등록 보류 (승인 대기 상태)
       │
       ├─ 자동으로 회사 관리자에게 승인 요청 발송
       │   (고객사, 포지션, 프리랜서 메시지 포함)
       │
       └─ 관리자 판단
            ├─ "다른 프로젝트입니다" → 등록 승인 → 신규 상태 진입
            ├─ "동일 프로젝트입니다" → 기존 프로젝트에 합류 안내
            ├─ "추가 확인 필요" → 메시지로 소통 후 결정
            └─ "반려" → 등록 거절 + 사유 전달
```


### 6.2 후보자 컨택 중복 방지

프로젝트가 등록된 후, 서칭·컨택 단계에서의 중복 방지.

**컨택 정책:**

| 상황 | 정책 |
|------|------|
| 같은 프로젝트, 같은 후보자 — 이미 다른 컨설턴트가 컨택함 | **차단** — 같은 잡 오퍼에 중복 연락 안 됨 |
| 다른 프로젝트, 같은 후보자 | **허용 + 알림** — 다른 잡 오퍼면 OK, 이전 컨택 이력 표시 |
| 같은 프로젝트, 같은 후보자 — 아직 컨택 안 했지만 예정 등록됨 | **경고** — "OOO가 컨택 예정으로 등록했습니다" |

**컨택 의사 선등록 단계:**

```
서칭 탭에서 후보자 선택
  → [컨택 예정 등록]  ← 여기서 잠금(lock) 발생
    → 실제 컨택 실행
      → [컨택 결과 기록]
```

#### 화면: 서칭 탭 — 충돌 표시

```
┌─ 서칭 탭: 후보자 추출 후 ──────────────────────────┐
│                                                    │
│  ☑ 홍길동  16년  품질부장                            │
│     ├─ 🔒 이 프로젝트: 전병권 컨택 예정 (04/06)      │  ← 같은 프로젝트 잠금
│     └─ ℹ️  프로젝트Y: 김소연 04/01 컨택 (관심 있음)  │  ← 다른 프로젝트 참조
│                                                    │
│  ☑ 김영희  14년  QA팀장                             │
│     └─ (컨택 이력 없음)                              │
│                                                    │
│  ☐ 박철수  18년  품질기획          ⚠ 노터치           │
│     └─ 🔒 이 프로젝트: 박준혁 컨택 예정 (04/05)      │  ← 다른 컨설턴트 선점
│                                                    │
│  [선택한 후보자 컨택 예정 등록 →]                     │
└────────────────────────────────────────────────────┘
```

**잠금 규칙:**
- 같은 프로젝트 내 선착순 — 컨택 예정 등록한 컨설턴트가 우선권
- 잠금 유효기간 — 예정 등록 후 7일 내 컨택 결과 미기록 시 자동 해제
- 수동 해제 — 프로젝트 리드가 잠금 해제 가능

## 8. 제출 서류 생성 파이프라인

현재 synco에 이미 있는 **이력서 → 데이터 추출** 파이프라인의 역방향. 추출된 데이터를 **고객사 제출용 문서로 가공**한다.

### 전체 흐름

```
① 후보자 DB 데이터 (이미 추출·검수 완료)
  │
  ② AI가 엑스다임 양식에 자동 매핑
  │   (오탈자, 문법, 서식, 날짜표기 통일)
  │
  ③ 컨설턴트 보완 — 상담 결과 추가 입력
  │   (텍스트 직접 입력 or 녹음 → 딕테이션 → AI 정리)
  │
  ④ AI가 최종 정리 — 완성본 생성
  │
  ⑤ 컨설턴트 최종 검토·수정
  │
  ⑥ 고객사 제출용 변환
      (개인정보 마스킹 + 포맷: Word/PDF + 국문/영문/국영문)
```

### ① → ② AI 초안 생성

## 9. 시스템 설계 원칙

전체 시스템에 관통하는 3가지 설계 원칙.

### 8.1 보이스 퍼스트 + 대화형 에이전트

모든 업무의 **입력·등록·진행은 대화형 에이전트가 주도**하고, 기존 탭/폼/칸반은 **조회·수정용**으로 보조한다.

**대화형 인터랙션 패턴:**

```
🎙️  "프로젝트 등록할거야"
🤖  고객사는 어느 회사인가요?
🎙️  "레이언스"
🤖  레이언스는 신규 고객사입니다. 웹에서 기업 정보를 검색할까요?
🎙️  "응"
🤖  레이언스 (Rayence) 찾았습니다.
    의료기기 | 코스닥 | 경기 화성 | 매출 2,800억
    맞나요? 맞으면 포지션을 알려주세요.
🎙️  "품질기획팀장. JD 파일 올릴게"
🤖  JD 분석했습니다. 이렇게 등록할까요?
    고객사: 레이언스 | 포지션: 품질기획팀장
    경력 15년+ | 인서울 | 품질경영기사
    [✅ 등록]  [✏️ 수정]  [🎙️ 수정사항 말하기]
🎙️  "등록해줘. 바로 서칭 시작하자"
🤖  ✅ 등록 완료. 후보자 8명 찾았습니다.
```

**원칙:**
- **컨텍스트 인식** — 현재 보고 있는 프로젝트/화면 기준으로 의도 파악
- **부족하면 질문** — 필수 정보 빠지면 에이전트가 하나씩 물어봄
- **항상 확인 후 저장** — 요약 보여주고 확인 받은 후에만 실행
- **다음 액션 제안** — 저장 후 자연스럽게 다음 단계 안내

### 8.2 업무 연속성 (컨시스턴시 유지)

업무 프로세스가 중단되었다가 재개될 때 **상태를 자동 보존**하고 이어서 진행.

```
ProjectContext (projects 앱)
  ├─ project (FK)
  ├─ consultant (FK → User)
  ├─ last_step ("contact_recording")
  ├─ pending_action ("홍길동 컨택 결과 입력 대기")
  ├─ draft_data (JSON — 작성 중이던 대화/폼 상태)
  └─ updated_at
```

**동작 방식:**

```
🎙️  "아까 하던 거 이어서 하자"
🤖  마지막 작업: Rayence 품질기획
    홍길동 컨택 결과 입력 중이었습니다.
    채널(전화/이메일/카톡)만 남았어요. 이어서 할까요?
```

프로젝트 진입 시에도 자동 표시:

```
┌─ Rayence 품질기획 ──────────────────────────────┐
│  ⏸️ 중단된 작업                                  │
│  홍길동 컨택 결과 입력 (3시간 전 중단)              │
│  [이어서 하기]  [새로 시작]  [취소]                │
└──────────────────────────────────────────────────┘
```

### 8.3 자동 생성 + 승인 요청

시스템이 **선제적으로 문서를 생성**하고 사용자에게 승인만 요청.

| 트리거 | 자동 생성물 | 알림 |
|--------|-----------|------|
| 프로젝트 등록 | 공지 초안 + 후보자 자동 서칭 결과 | "공지/후보자 준비됨" |
| 컨택 → 관심 있음 | 제출 서류 AI 초안 | "서류 초안 검토 요청" |
| 서류 제출 완료 | 고객사 팔로업 리마인더 예약 | "3일 후 팔로업 알림 예정" |
| 면접 합격 | 오퍼 템플릿 | "오퍼 초안 준비됨" |
| 잠금 유효기간 임박 | 재컨택 리마인더 | "컨택 잠금 내일 만료" |

## 10. 텔레그램 채널 통합

앱 푸시 대신 **텔레그램을 메인 알림·업무 채널**로 사용. Bot API Inline Keyboard로 **버튼 기반 객관식 업무 처리** 지원.

### 채널 역할 분담

| 채널 | 역할 | 언제 |
|------|------|------|
| **텔레그램** | 알림, 승인, 상태 확인, 빠른 업무 입력 | 이동 중, 즉시 응답 |
| **웹 앱** | 상세 조회, 서류 편집, 칸반 보드, 데이터 비교 | 심층 작업 |

텔레그램에서 시작 → 복잡한 건 "상세 보기 →" 링크로 웹 앱 연결.

### 알림 + 승인 (Inline Keyboard)

```
🤖 [synco] 프로젝트 등록 승인 요청

   전병권 → Rayence 품질기획팀장
```

### 11.2 기업 DB

**사전 구축하지 않는다.** 국내 상장사(KOSPI/KOSDAQ) 리스트만 초기 데이터로 보유.

```
CompanyProfile (clients 앱)
  ├─ name ("레이언스")
  ├─ name_en ("Rayence")
  ├─ industry ("의료기기")
  ├─ size_category (대기업/중견/중소/외국계/스타트업)
  ├─ revenue_range, employee_count_range
  ├─ listed (KOSPI/KOSDAQ/비상장/해외상장)
  ├─ region ("경기 화성")
  └─ notes
```

**점진적 구축:** 프로젝트 등록 시 고객사 입력 → DB에 없으면 웹 검색 → 자동 채움 → 저장.

### 11.3 자격증 DB

**최대한 풍부하게 초기 데이터 구축.** 이력서 파싱 자동 매칭에도 활용.

```
PreferredCert (clients 앱)
  ├─ name ("KICPA")
  ├─ full_name ("한국공인회계사")
  ├─ category (회계/법률/기술/안전/품질/의료/IT/무역/건설/식품/어학/...)
  ├─ level (상/중/하)
  └─ aliases (JSON: ["CPA", "공인회계사", "KICPA"])
```

**초기 데이터 소스:**
- 한국산업인력공단 국가자격증 전체 목록 (~800종)
- 민간자격증 주요 목록
- 국제 자격증 (CPA, CFA, PMP, AWS, CISSP 등)

| 카테고리 | 예시 | 규모 |

| 기술/엔지니어링 | 기술사, 기사 (산업안전, 품질경영, 전기, 화학 등) | ~200종 |

## 15. 이메일 이력서 자동 처리

잡포털 공지 후 이메일로 수신되는 이력서 자동 처리.

### 처리 플로우

```
이메일 수신
  → 첨부파일 추출 (PDF, Word, HWP)
  → 프로젝트 매칭 (제목/본문 키워드 or 레퍼런스 코드)
  → data_extraction 파이프라인으로 파싱
  → 후보자 DB 등록 + 프로젝트 연결
  → 컨설턴트에게 텔레그램 알림
```

### 텔레그램 알림

```
🤖 [synco] 📬 새 지원자 2명 — Rayence 품질기획

   1. 박영수 | 15년 | 오스템 품질부장 | 적합도: 높음
   2. 김민지 | 8년 | 바텍 QA팀 | ⚠️ 경력 부족

   ┌──────────────────────────┐
   │ 👁️ 상세 보기 (웹)        │
   ├──────────────────────────┤
   │ 📞 박영수 컨택 예정 등록   │
   ├──────────────────────────┤
   │ ⏸️ 김민지 보류            │
   └──────────────────────────┘
```

### 연동 방식 (단계적)

| 단계 | 방식 | 설명 |
|------|------|------|
| **1단계** | 수동 업로드 | 프로젝트 서칭 탭에 드래그 앤 드롭 |
| **2단계** | Gmail API | 지정 이메일 자동 모니터링 (Google OAuth 활용) |
