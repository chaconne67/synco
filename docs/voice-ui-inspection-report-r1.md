## 상세 점검 결과

### 3.1 플로팅 버튼 / 진입점

#### 항목 1.1 — 아이콘이 음성 검색을 나타내는가

- **코드 확인:** `search.html:98-101` — SVG path가 말풍선 + 점 세 개 (채팅 아이콘)
  ```
  d="M8 12h.01M12 12h.01M16 12h.01M21 12c0 4.418-4.03 8-9 8a9.863..."
  ```
- **판정: ❌ 문제**
- **근거:** 채팅 말풍선 아이콘은 "텍스트 채팅"을 연상시킴. 보이스 퍼스트 서비스에서 플로팅 버튼이 마이크 아이콘이어야 사용자가 음성 검색 기능을 즉시 인지함. Google Voice Search, ChatGPT 모두 마이크 아이콘을 CTA로 사용.
- **수정 방향:** SVG를 마이크 아이콘으로 교체. `chatbot_modal.html:47-49`에 이미 마이크 SVG가 있으므로 동일한 path 재사용 가능.

#### 항목 1.2 — 버튼에 레이블/힌트가 있는가

- **코드 확인:** `search.html:97` — `aria-label="AI 검색 열기"`만 존재. 시각적 텍스트/툴팁/배지 없음.
- **판정: ❌ 문제**
- **근거:** aria-label은 스크린 리더 전용이며 시각적으로 보이지 않음. 첫 방문 사용자가 버튼의 용도를 아이콘만으로 파악해야 함. 리서치에 따르면 140자 이하 짧은 툴팁으로 기능 안내가 권장됨.
- **수정 방향:** 첫 방문 시 "음성으로 검색해보세요" 툴팁 또는 텍스트 배지 표시. `localStorage`로 표시 여부 관리.

#### 항목 1.3 — 버튼 크기/위치가 터치하기 쉬운가

- **코드 확인:** `search.html:93-96`
  - 모바일: `w-14 h-14` = 56px (WCAG 최소 48px 충족)
  - 데스크톱: `lg:w-[60px] lg:h-[60px]` = 60px
  - 위치: `fixed bottom-6 right-4 lg:right-6`
  - `search.html:8-9`에서 `{% block sidebar %}{% endblock %}`, `{% block bottom_nav %}{% endblock %}`로 사이드바/하단 네비 제거 — 겹침 없음
- **판정: ✅ 통과**
- **근거:** 56px은 최소 48px 기준 충족. `bottom-6 right-4` (24px, 16px)로 화면 모서리에서 충분한 여백. 하단 네비가 없어 겹침 우려 없음.

#### 항목 1.4 — 검색 상태가 버튼에 반영되는가

- **코드 확인:** `search.html:91-102` — 버튼은 항상 동일한 `bg-primary text-white` + 말풍선 아이콘. 검색 세션 활성 여부와 무관하게 외관 변화 없음. `chatbot.js`에서도 `chatbot-toggle` 버튼의 외관을 변경하는 코드 없음.
- **판정: ⚠️ 개선 권장**
- **근거:** 이전 검색 결과가 있을 때(세션 활성 시) 시각적 구분이 없어, 사용자가 이전 검색 맥락의 존재를 인지하기 어려움.
- **수정 방향:** 활성 세션이 있을 때 버튼에 작은 도트 배지(예: 녹색 3px 원) 추가.

### 3.2 마이크 아이콘 & 상태 표시

#### 항목 2.1 — 마이크 버튼이 입력바에서 시각적으로 우선하는가

- **코드 확인:** `chatbot_modal.html:42-71` — 입력바 레이아웃:
  ```
  [마이크 40px] [텍스트 입력 flex-1] [전송 40px]
  ```
  텍스트 입력이 `flex-1`로 가용 공간 전체를 차지. 마이크와 전송 버튼은 동일 크기(40px)의 보조 요소.
- **판정: ❌ 문제**
- **근거:** 보이스 퍼스트 서비스에서 텍스트 입력창이 시각적으로 가장 큰 요소. 사용자 시선이 텍스트 입력으로 먼저 향함. 2026 AI 앱 트렌드 리서치에 따르면 마이크를 메인 CTA로 배치하고 텍스트는 보조/폴백으로 축소해야 함.
- **수정 방향:** 마이크 버튼을 중앙에 크게 배치(56-64px), 텍스트 입력은 축소하거나 "텍스트로 입력" 토글로 변경.

#### 항목 2.2 — idle 상태 아이콘이 명확한가

- **코드 확인:** `chatbot_modal.html:47-49` — 마이크 SVG 윤곽선 아이콘, `w-5 h-5` (20px). 버튼 자체는 `w-10 h-10` (40px) 원형 + `bg-primary text-white`.
- **판정: ✅ 통과**
- **근거:** 마이크 아이콘은 명확하고 인지 가능. 40px 원형 버튼에 bg-primary 배경색으로 탭 가능함이 전달됨. 다만 보이스 퍼스트 전환 시 크기 확대 필요(별도 항목 2.1에서 다룸).

#### 항목 2.3 — recording 상태가 idle과 확실히 구분되는가

- **코드 확인:**
  - `chatbot_modal.html:51-53` — recording 아이콘: `<rect x="6" y="6" width="12" height="12" rx="2"/>` (둥근 정사각형 = 정지 아이콘)
  - `chatbot.js:209-211` — `bg-primary` → `bg-red-500`으로 교체, `recIcon` 표시
- **판정: ❌ 문제**
- **근거:** 색상 변화(파란→빨간)와 아이콘 변경(마이크→정사각형)은 있으나, 리서치에서 권장하는 핵심 피드백인 펄스 애니메이션/파형이 전혀 없음. 정사각형 아이콘은 "정지" 의미이나, 실시간으로 "녹음 중"임을 보여주는 동적 피드백이 부재. Medium 리서치에 따르면 녹음 중 깜빡임(blink) + 파형 시각화가 가장 효과적.
- **수정 방향:** `animate-pulse` CSS 적용 + 마이크 버튼 주변 파형/ring 애니메이션 추가. 최소 구현으로 `animate-pulse` 클래스만 추가해도 개선.

#### 항목 2.4 — processing 상태가 구분되는가

- **코드 확인:**
  - `chatbot_modal.html:54` — `<span id="mic-spinner" class="hidden btn-spinner"></span>`
  - `chatbot.js:212-214` — `bg-gray-400` + spinner 표시
  - `base.html:54` — `btn-spinner` 정의: 1em 크기 회전 스피너
- **판정: ❌ 문제**
- **근거:** 스피너 자체는 적절하나, "음성 인식 중..." 같은 텍스트 보조가 없음. 사용자는 스피너만으로 현재 무엇을 처리 중인지(STT vs 검색) 구분할 수 없음. VUI 3초 규칙에 따르면 각 단계에서 명확한 상태 메시지가 필요.
- **수정 방향:** 마이크 버튼 아래 또는 채팅 영역에 "음성 인식 중..." 텍스트 표시. `appendThinking()` 함수를 STT 단계에서도 활용하되 메시지를 "음성 인식 중..."으로 구분.

#### 항목 2.5 — 상태 전환이 부드러운가

- **코드 확인:** `chatbot.js:196-216 setMicState()` — `btn.className.replace()` 방식으로 CSS 클래스를 문자열 치환. 아이콘은 `classList.add/remove("hidden")`으로 즉시 표시/숨김 전환.
  - `chatbot_modal.html:45` — 버튼에 `transition` 클래스 있음 (Tailwind `transition` = `transition-property: all; duration: 150ms`)
- **판정: ⚠️ 개선 권장**
- **근거:** 버튼 자체에 `transition` 클래스가 있어 배경색 전환은 150ms fade가 적용됨. 그러나 아이콘 교체는 `hidden` 클래스의 `display:none` 토글이므로 즉시 전환(트랜지션 없음). 아이콘 간 전환 시 scale/fade 효과가 있으면 더 매끄러움.
- **수정 방향:** 아이콘 전환에 `opacity` + `scale` 트랜지션 적용. 또는 단일 아이콘 요소에서 SVG path를 morph하는 방식.

### 3.3 녹음 중 인디케이터 (파형, 애니메이션 등)

#### 항목 3.1 — 녹음 중 실시간 시각적 피드백이 있는가

- **코드 확인:** `chatbot.js` 전체 + `chatbot_modal.html` 전체 검토 — 녹음 중 시각적 피드백은 버튼 배경색 변경(`bg-red-500`)과 정사각형 아이콘 표시뿐. 파형, 펄스, 오디오 반응형 애니메이션 코드 없음.
- **판정: ❌ 문제**
- **근거:** 사용자가 말하는 동안 UI가 "듣고 있음"을 시각적으로 표현하지 않음. Google Voice Search는 실시간 dynamic arc 파형, ElevenLabs는 Canvas 기반 waveform을 사용. 최소한 CSS `animate-pulse`라도 적용해야 함.
- **수정 방향:**
  - **최소:** 녹음 중 `animate-pulse` + `ring-4 ring-red-500/30` 적용
  - **권장:** Web Audio API `AnalyserNode`로 실시간 음량 → CSS 변수로 전달 → 파형 크기 반응
  - **참조:** wavesurfer.js, ElevenLabs Waveform UI 컴포넌트

#### 항목 3.2 — 파형이 마이크 입력에 반응하는가

- **코드 확인:** `chatbot.js` — `MediaRecorder` 사용 시 `stream`을 받지만, Web Audio API `AudioContext`나 `AnalyserNode` 연결 코드 없음. `stream`은 `onstop`에서 `getTracks().stop()`으로 해제됨.
- **판정: ❌ 문제**
- **근거:** 미구현. 실시간 오디오 반응형 파형이 없으므로 사용자는 마이크가 실제로 음성을 수신하고 있는지 확인할 수 없음.
- **수정 방향:** `startRecording()`에서 `AudioContext` + `AnalyserNode` 생성 → `requestAnimationFrame` 루프로 음량 데이터 추출 → CSS custom property 또는 Canvas로 시각화.

#### 항목 3.3 — 녹음 시간/경과가 표시되는가

- **코드 확인:** `chatbot.js` + `chatbot_modal.html` — 녹음 시간 카운터나 타이머 관련 코드 없음. `setInterval`이나 경과 시간 표시 로직 부재.
- **판정: ❌ 문제**
- **근거:** 사용자가 얼마나 녹음 중인지 알 수 없음. 최대 녹음 시간 제한도 없어(서버의 10MB 제한만 존재: `views.py:370`), 사용자가 너무 긴 녹음을 할 수 있음.
- **수정 방향:** 녹음 시작 시 `setInterval`로 경과 시간 표시 (00:03 형태). 최대 녹음 시간(예: 30초) 설정 + 자동 정지.

#### 항목 3.4 — reduced-motion 설정 시 대체 인디케이터가 있는가

- **코드 확인:** `input.css:16-22` — `prefers-reduced-motion: reduce` 미디어 쿼리 적용됨:
  ```css
  animation-duration: 0.01ms !important;
  animation-iteration-count: 1 !important;
  transition-duration: 0.01ms !important;
  ```
- **판정: ⚠️ 개선 권장**
- **근거:** 현재 녹음 중 애니메이션 자체가 없으므로 이 항목은 미래 구현에 해당. 다만 `reduced-motion` 미디어 쿼리는 이미 전역 적용되어 있어, 파형 애니메이션 추가 시 자동으로 비활성화됨. 이 경우 정적 "녹음 중" 텍스트 + 색상 인디케이터가 대체 피드백으로 필요.
- **수정 방향:** 파형 애니메이션 구현 시 `reduced-motion` 상태에서는 "녹음 중" 텍스트 + 정적 빨간 배경으로 대체.

### 3.4 음성 입력 vs 텍스트 입력 UX

#### 항목 4.1 — 음성이 주 입력, 텍스트가 보조인 레이아웃인가

- **코드 확인:** `chatbot_modal.html:41-71` — 입력바:
  ```html
  <button id="mic-btn" class="w-10 h-10 ...">  <!-- 40px -->
  <input id="chat-input" class="flex-1 ...">    <!-- 나머지 공간 전체 -->
  <button class="w-10 h-10 ...">               <!-- 전송 40px -->
  ```
  텍스트 입력이 `flex-1`로 대부분의 공간 차지. placeholder: "텍스트로 입력하세요..."
- **판정: ❌ 문제**
- **근거:** 전형적인 텍스트 채팅 레이아웃. 마이크는 보조 버튼 위치. ChatGPT 2025.11 통합 업데이트, 2026 AI 앱 트렌드 모두 "음성이 기본, 텍스트가 폴백" 패턴을 권장. 현재 정반대 구조.
- **수정 방향:** 마이크 버튼을 중앙 대형(56-64px)으로 배치. 텍스트 입력은 "키보드로 입력" 링크/작은 버튼으로 축소하거나, 마이크 버튼 아래에 작게 배치.

#### 항목 4.2 — 음성↔텍스트 전환이 매끄러운가

- **코드 확인:**
  - `chatbot.js:34-43 sendMessage()` — 텍스트 전송
  - `chatbot.js:45-51 toggleRecording()` — 음성 녹음 토글
  - `chatbot.js:110-145 doSearch()` — 검색 실행 시 `inputType` 파라미터만 다름
  - 세션(`sessionId`)이 공유되어 같은 대화 스레드에서 음성/텍스트 혼용 가능
- **판정: ⚠️ 개선 권장**
- **근거:** 기능적으로는 같은 세션에서 음성/텍스트 전환이 가능하고 컨텍스트가 유지됨 (세션 기반 필터 누적). 그러나 UI 상 음성과 텍스트가 별개의 입력 경로처럼 보임. ChatGPT 스타일의 "같은 인터페이스에서 자연스럽게 전환" 경험과는 차이가 있음.
- **수정 방향:** 입력 모드 전환 시 시각적 피드백 추가 (예: 텍스트 입력 클릭 시 부드러운 확장 애니메이션).

#### 항목 4.3 — 실시간 트랜스크립션이 표시되는가

- **코드 확인:** `chatbot.js:82-108 handleRecordingComplete()` — 녹음 완료 후 Blob을 서버(`/candidates/voice/`)로 전송 → Whisper API 처리 → 결과 텍스트를 `appendUserMessage(data.text, true)`로 한 번에 표시. 녹음 중 또는 처리 중 텍스트 변환 결과를 보여주지 않음.
- **판정: ❌ 문제**
- **근거:** Whisper API는 일괄 처리(batch) 방식이므로 완료까지 텍스트가 보이지 않음. 사용자는 자신이 뭐라고 말했는지, 정확히 인식되었는지 확인할 기회가 없음. 2026 트렌드에 따르면 실시간 텍스트 변환 표시 → 제출 전 사용자 수정 가능이 권장됨.
- **수정 방향:**
  - **단기:** Whisper 결과 표시 후 "수정" 버튼 추가 → 텍스트 수정 후 재검색
  - **장기:** Web Speech API(`SpeechRecognition`)로 실시간 중간 결과(interim results) 표시 → 최종 결과는 Whisper로 확정

#### 항목 4.4 — 모달 열림 시 포커스가 적절한가

- **코드 확인:** `chatbot.js:26` — `document.getElementById("chat-input").focus();` — 모달 열릴 때 텍스트 입력에 자동 포커스.
- **판정: ❌ 문제**
- **근거:** 모바일에서 `input.focus()`는 가상 키보드를 자동으로 올림. 보이스 퍼스트 서비스에서 모달을 열었을 때 키보드가 화면의 절반을 가리면 마이크 버튼 접근성이 크게 저하됨. 사용자가 음성으로 검색하려 했는데 키보드가 올라오는 것은 역효과.
- **수정 방향:** `chat-input.focus()` 제거. 보이스 퍼스트라면 포커스를 마이크 버튼이나 "탭하여 말하기" 안내 요소로 이동. 텍스트 입력은 사용자가 직접 탭할 때만 포커스.

### 3.5 대화 플로우 (음성 → 결과 표시)

#### 항목 5.1 — 음성 입력 후 결과까지 피드백 체인이 끊기지 않는가

- **코드 확인:** `chatbot.js` 전체 플로우:
  1. `startRecording()` → `setMicState("recording")` — 버튼 빨간색
  2. `stopRecording()` → `handleRecordingComplete()` → `setMicState("processing")` — 스피너
  3. Whisper 응답 → `setMicState("idle")` + `appendUserMessage()` + `appendThinking()` — "생각 중..." 표시
  4. `doSearch()` 응답 → `removeThinking()` + `appendAIMessage()` — 결과 표시
- **판정: ⚠️ 개선 권장**
- **근거:** 기본적인 피드백 체인은 존재하나, 단계별 메시지가 불충분. 특히:
  - 녹음 중: 빨간 버튼뿐, "듣고 있습니다" 메시지 없음
  - processing: 스피너뿐, "음성 인식 중..." 텍스트 없음
  - Whisper → 검색 사이 "검색 중..." 메시지는 `appendThinking()`의 "생각 중..."으로 대체
  - VUI 3초 규칙: Whisper API 지연(보통 2-5초)과 검색 지연(1-3초)이 합산되면 5-8초 무응답 가능
- **수정 방향:** 각 단계에 명시적 상태 메시지 추가: "듣고 있습니다..." → "음성 인식 중..." → "검색 중..." → 결과.

#### 항목 5.2 — AI 응답이 음성에 최적화되어 있는가

- **코드 확인:** `candidates/views.py:325-327`
  ```python
  ai_message = parsed.get("ai_message", "")
  if not ai_message:
      ai_message = f"{result_count}명의 후보자를 찾았습니다."
  ```
  `parse_search_query()`가 `ai_message`를 반환하면 사용하고, 없으면 기본 메시지.
- **판정: ✅ 통과**
- **근거:** 기본 메시지 `"{N}명의 후보자를 찾았습니다."`는 짧고 핵심적. VUI "One Breath" 규칙에 부합. LLM이 생성하는 `ai_message`의 길이는 프롬프트에 따라 다르지만, 기본 구조는 적절.

#### 항목 5.3 — 결과 표시 시 모달↔리스트 연계가 자연스러운가

- **코드 확인:** `chatbot.js:238-242 refreshCandidateList()` — HTMX ajax로 `#candidate-list` 업데이트. 모달은 열린 상태로 유지. 사용자가 결과를 보려면 모달을 수동으로 닫아야 함(`toggleChatbot()`).
  - `chatbot.js:17-21` — 모달 닫을 때 `refreshCandidateList()` 호출 → 리스트 갱신
- **판정: ⚠️ 개선 권장**
- **근거:** 검색 결과가 모달 뒤의 리스트에 업데이트되지만, 모달이 모바일에서 화면 85%를 차지(`h-[85vh]`)하므로 리스트가 거의 보이지 않음. 사용자가 결과를 보려면 모달을 닫는 추가 동작 필요.
- **수정 방향:** 검색 결과 수신 후 모달을 자동으로 최소화(status bar 크기로 축소)하거나, 모달 내에서 결과 미리보기를 제공.

#### 항목 5.4 — 검색 상태바가 충분한 정보를 제공하는가

- **코드 확인:** `search_status_bar.html:1-15`

- **근거:** 검색어와 결과 수를 간결하게 표시하며 클릭으로 모달 재진입 가능. 다만 적용된 필터 상세(지역, 나이, 성별 등)나 클리어 버튼은 없음.

### 3.6 에러 상태 (마이크 권한 거부, 네트워크 오류 등)

#### 항목 6.1 — 마이크 권한 거부 시 대응이 충분한가

- **코드 확인:** `chatbot.js:70-72`

- **수정 방향:** 토스트 대신 인라인 안내 표시:
  - "마이크 권한이 거부되었습니다"
  - "브라우저 주소창 왼쪽 🔒 아이콘 → 마이크 → 허용으로 변경해주세요"
  - "텍스트로도 검색할 수 있습니다" + 텍스트 입력 포커스

#### 항목 6.2 — 음성 인식 실패 시 대응이 충분한가

- **코드 확인:** `chatbot.js:104-107`
  ```javascript

    setMicState("idle");
  ```

- **근거:** 실패 원인(네트워크 오류 vs 무음 vs 인식 불가)을 구분하지 않음. VUI 설계 원칙에 따르면 맥락적 재프롬프트가 필요하고, 2회 실패 시 탈출 경로(텍스트 전환) 자동 제안이 권장됨.

#### 항목 6.3 — 네트워크 오류 시 대응

- **코드 확인:** `chatbot.js:140-144`

- **판정: ❌ 문제**
- **근거:** "검색 중 오류가 발생했습니다"는 원인을 알려주지 않음. `navigator.onLine` 체크나 재시도 버튼이 없음. 사용자가 다음 행동을 결정하기 어려움.
- **수정 방향:** `navigator.onLine === false` 시 "인터넷 연결을 확인해주세요" 메시지 + 연결 복구 시 자동 재시도. 온라인이면 "일시적 오류입니다. 다시 시도해주세요." + 재시도 버튼.

#### 항목 6.4 — Rate limit 초과 시 대응

- **코드 확인:**
  - `candidates/views.py:273-276` — 검색: 10회/60초, status=429
  - `candidates/views.py:361-363` — 음성: 5회/60초, status=429

- **판정: ❌ 문제**
- **근거:** "잠시 후"가 정확히 얼마인지 알 수 없음. 남은 대기 시간이나 쿨다운 타이머 없음. 사용자가 반복 시도하여 추가 좌절 유발 가능.
- **수정 방향:** 429 응답에 `retry_after` 필드 포함 → 클라이언트에서 남은 시간 카운트다운 표시 + 마이크/전송 버튼 임시 비활성화.

#### 항목 6.5 — 무음 녹음 (아무 말 안 함) 시 대응

  - `whisper.py:33-40` — Whisper API에 그대로 전달, 빈 텍스트 반환 가능

  - 이 에러가 `doSearch` 응답의 `data.error`로 돌아와 `appendAIMessage`에 표시됨
- **판정: ⚠️ 개선 권장**
- **근거:** 무음 시 흐름: 녹음 → Whisper(빈 텍스트) → `appendUserMessage("")` + 검색 요청 → "메시지를 입력해주세요" 에러. 사용자 메시지로 빈 텍스트가 표시되고 "메시지를 입력해주세요"는 음성 컨텍스트에 부적합. "음성이 감지되지 않았습니다"가 더 적절.

  - 마이크 아이콘으로 교체 (1.1)
  - 첫 방문 시 "음성으로 검색해보세요" 툴팁 표시
  - `localStorage.getItem("onboarding_seen")` 체크로 1회만 표시

#### 항목 8.2 — 모달 내 사용 가이드가 있는가

- **코드 확인:** `chatbot_modal.html:32-37` — 기본 인사말만:
  ```html
  <p class="text-[15px] text-gray-900">안녕하세요! 어떤 후보자를 찾으시나요?</p>
  ```
  `chat_messages.html:1-5` — 동일한 인사말. 마이크 사용법, 예시 검색어 안내 없음.
- **판정: ❌ 문제**
- **근거:** 첫 사용자가 "어떤 후보자를 찾으시나요?"만 보면 텍스트로 입력하려 할 것. 마이크 버튼 존재를 인지하지 못하거나, 어떻게 말해야 하는지(검색 문법/예시) 모름. 리서치에 따르면 예시 검색어 제시가 중요.
- **수정 방향:** 인사말 개선:
  - "마이크를 눌러 말씀하세요"
  - 예시 칩: "강남구 20대 여성", "서울대 출신 경력 10년", "영어 가능한 30대"
  - 칩 클릭 시 해당 검색어로 자동 검색

#### 항목 8.3 — 마이크 권한 요청 전 사전 안내가 있는가

- **코드 확인:** `chatbot.js:53-72 startRecording()` — `navigator.mediaDevices.getUserMedia({ audio: true })`를 바로 호출. 사전 안내(pre-permission) UI 없음.
- **판정: ❌ 문제**
- **근거:** 리서치에 따르면 시스템 권한 프롬프트 전에 사전 안내를 하면 승인율이 12% → 30%+ 로 향상. 현재는 마이크 버튼 클릭 즉시 브라우저 권한 팝업이 뜨므로 사용자가 맥락 없이 결정해야 함.
- **수정 방향:** 첫 마이크 클릭 시 pre-permission 모달:
  - "음성 검색을 위해 마이크 접근이 필요합니다"
  - "마이크로 말씀하시면 AI가 후보자를 찾아드립니다"
  - [허용하기] [텍스트로 검색하기] 버튼
  - `localStorage`로 1회만 표시

#### 항목 8.4 — 검색 가능한 항목의 예시가 제공되는가

- **코드 확인:** `chatbot_modal.html` + `search.html` 전체 — 검색 예시 칩, 카드, 제안 텍스트 없음. 카테고리 탭(`search.html:44-77`)은 있으나 이는 필터 탭이지 검색 예시가 아님.
- **판정: ❌ 문제**
- **근거:** 사용자가 어떤 검색어를 말할 수 있는지 모름. "20대 여성 강남" 같은 자연어를 지원하는지, "서울대 출신"을 이해하는지 사전 정보 없음. 리서치에 따르면 검색 예시 칩/카드 제공이 사용자 첫 사용 장벽을 크게 낮춤.
- **수정 방향:** 모달 내 인사말 아래에 예시 검색 칩 배치:
  ```
  "강남 30대 남성" | "서울대 출신" | "경력 10년 이상" | "영어 가능"
  ```
  칩 클릭 시 해당 텍스트로 검색 실행.

#### 항목 8.5 — 온보딩을 다시 볼 수 있는가

- **코드 확인:** `chatbot_modal.html` — 도움말/? 버튼 없음. 온보딩 자체가 없으므로 재표시 기능도 없음.
- **판정: ⚠️ 개선 권장**
- **근거:** 온보딩 구현 후에 필요한 기능. 현재는 온보딩 자체가 없으므로 우선순위 낮음. 온보딩 구현 시 함께 추가.
- **수정 방향:** 모달 헤더에 `?` 또는 도움말 아이콘 → 사용 가이드 + 예시 재표시.

## 종합 소견

현재 synco 보이스 서치 UI는 **기능적으로는 음성 검색이 동작하나, UX는 전형적인 텍스트 채팅 인터페이스**에 머물러 있음. "보이스 퍼스트"로 전환하기 위해 가장 시급한 3가지:

1. **진입점 아이콘 교체** — 채팅 말풍선 → 마이크 (사용자가 음성 기능을 발견할 수 있게)
2. **입력바 레이아웃 역전** — 마이크를 중앙 대형 CTA로, 텍스트를 보조로 (음성이 기본임을 전달)
3. **녹음 중 시각 피드백** — 최소 pulse 애니메이션이라도 추가 (시스템이 듣고 있음을 확인)

이 3가지만 수정해도 보이스 퍼스트 인식이 크게 개선됨. 나머지 항목은 우선순위 순서대로 점진적으로 개선 가능.
