## Problem Statement

어제 APPROVED된 DB-First 디자인(이력서 파싱 파이프라인 + 검색 UI)의 Phase 2를 구체화하는 세션. 핵심 질문: 20만 건의 구조화된 후보자 DB를 헤드헌터가 어떻게 검색하고 사용하는가?

기존 synco v1(FC CRM)의 인터페이스와 기능을 전면 개편하여, 헤드헌팅 업무에 특화된 음성 중심 검색 도구로 재설계한다.

## Demand Evidence

(어제 디자인 doc에서 이미 검증됨)
- 20년차 헤드헌터가 직접 의뢰. 업체당 ~1,000만원 지불 의향.
- 국내 활성 서치펌 ~1,000곳이 잠재 고객.
- 친구분의 업계 인맥을 통한 추가 영업 기회.

## Constraints

- 기존 synco Django 5.2 + HTMX + Tailwind + PostgreSQL + pgvector 코드베이스 활용
- Pretendard 폰트, synco indigo (#5B6ABF) 디자인 시스템 유지
- Phase 1(이력서 파싱 파이프라인)이 선행 완료되어야 실 데이터로 검색 가능
- 음성 인식: Whisper API 우선 (한국어 전문용어 정확도 보장)
- 대화형 multi-turn 검색은 MVP 필수

## Premises

1. 음성 대화형 검색이 MVP 핵심 차별화이며, 텍스트 검색은 동급 fallback이다.
2. 기존 v1 기능(연락처 CRM, 미팅 관리, 대시보드, AI 브리핑 등)은 v2에서 전부 제거하고, DB 추출 + 검색만 남긴다.
3. 데이터 검증(파싱 결과 확인/수정)은 시각적 UI가 필요하고, 음성으로 대체할 수 없다.
4. **REVISED (Codex 도전 반영):** OpenAI Whisper API로 음성 인식. Web Speech API는 fallback. 한국어 전문용어(회사명, 학교명, 직무명) 정확도가 80%+ 나와야 음성이 차별화가 됨.
5. 대화 맥락(multi-turn)은 서버에서 LLM이 관리하며, 이전 검색 조건을 기억해서 점진적으로 좁혀나간다.

## Cross-Model Perspective

**Codex cold read:**
- Steelman: "This is not an ATS. It's a private talent-retrieval engine for Korean search firms. Turn 200K dead files into a searchable institutional asset."
- Key insight: "Search = folder browsing + memory" is the whole business. The product is **memory externalization**: junior staff get senior-level search quality.
- Challenged premise #4: Web Speech API won't handle Korean professional terms (company names, schools, job titles mixed with English). First-query success needs 80%+, otherwise voice is a demo gimmick. → **ACCEPTED, premise revised to Whisper API.**
- 48-hour prototype: One search page with mic + text fallback. LLM converts each turn into structured filters + semantic query. Hybrid retrieval. Result list with reason snippets.

## Landscape Awareness

**Voice AI 시장:** $7.6B(2025) → $139B(2033). VC 투자 7배 증가 ($315M→$2.1B).

**기존 AI 리크루팅 도구들의 두 갈래:**
1. 공개 데이터 소싱 (Juicebox/PeopleGPT: 8억+ 공개 프로필 자연어 검색)
2. 지원자 대면 자동화 (Paradox: 대화형 채용 챗봇)

**synco의 차별화:** 둘 다 아님. 헤드헌터의 **사적 DB**(20만 건 자체 보유 파일)를 음성 대화형으로 검색. "공개 데이터 소싱"도 "지원자 대면 자동화"도 아닌, **"서치펌의 자산을 열어주는 도구"**.

아무도 이 조합(사적 DB + 음성 자연어 검색 + 한국어)을 하고 있지 않음.

## Approaches Considered

### Approach A: Voice Chat Page (단일 채팅 페이지)
- **Summary:** 상단 대화, 하단 결과 리스트. 카카오톡 스타일.
- **Effort:** S (CC: ~3-4일)
- **Risk:** Low
- **Pros:** 최소 시간, 익숙한 채팅 UX
- **Cons:** 결과 20명이 대화 사이에 끼면 스크롤 길어짐, 비교 불편

### Approach B: Floating Chatbot + Full Results (Selected)
- **Summary:** 전체 화면 후보자 리스트 + 우하단 플로팅 챗봇 버튼. 챗봇 클릭 → 모달로 대화형 검색. 검색 완료 → 최소화 → 결과 확인. 다시 열면 이전 대화 유지.
- **Effort:** S-M (CC: ~4-5일)
- **Risk:** Low
- **Pros:** 결과에 전체 화면 사용 (비교/스캔 최적), 채팅은 필요할 때만, 대화 컨텍스트 유지, Intercom 스타일로 익숙
- **Cons:** 모달 UX 세부 조정 필요 (모바일에서 크기, 위치)

### Approach C: Conversational Agent (AI 주도 대화)
- **Summary:** AI가 능동적으로 질문하고 추천 이유를 설명. "비서" 경험.
- **Effort:** L (CC: ~2주)
- **Risk:** Med-High
- **Pros:** WOW 팩터 극대화, junior→senior 검색 품질
- **Cons:** LLM 다중 호출, 응답 지연, 프롬프트 복잡도

## Recommended Approach

**Approach B: Floating Chatbot + Full Results.**

이유: 결과 리스트에 전체 화면을 쓸 수 있어서 비교/스캔에 최적. 챗봇은 필요할 때만 열리고 안 쓸 때 화면을 차지하지 않음. Intercom/카카오 채널 패턴으로 한국 사용자에게 이미 익숙. 4-5일이면 배포 가능하고, 추후 C(AI 주도 대화)로 확장 용이.

## Interface Design

### 화면 구성 (3개 State)

#### State 1: 메인 화면 (검색 결과 + 플로팅 버튼)

```
┌─────────────────────────────────────┐
│  synco                              │  ← 헤더 (로고)
├─────────────────────────────────────┤
│  [전체] [Accounting] [EHS] [...]    │  ← 카테고리 탭 (20개, 스크롤)
├─────────────────────────────────────┤
│  🔍 "회계 10년차 이상" — 8명 찾음    │  ← 검색 상태 바
├─────────────────────────────────────┤
│  ┌───────────────────────────────┐  │
│  │ 강솔찬 | 현대엠시트 | 12년    │  │  ← 후보자 카드
│  │ 회계팀장 · 현직               │  │     (이름, 회사, 경력, 태그)
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │ 김영희 | 삼성전자 | 15년      │  │
│  │ 재무본부 · 현직               │  │
│  └───────────────────────────────┘  │
│  ... (무한 스크롤)                   │
│                                     │
│                          ┌─────┐    │
│                          │ 💬  │    │  ← 플로팅 챗봇 버튼
│                          └─────┘    │
└─────────────────────────────────────┘
```

**카테고리 탭 동작:**
- 전체: 모든 카테고리
- 개별 탭 클릭: 해당 카테고리 필터. 카테고리 탭은 폴더 구조에 익숙한 파워유저를 위한 시각적 앵커.
- **탭-음성 상호작용:** 음성 쿼리에서 LLM이 카테고리를 추출하면, 탭이 자동으로 해당 카테고리로 전환됨. 음성이 주 인텐트, 탭은 보조 네비게이션. "회계 경력자 찾아줘" 시 Sales 탭이 선택되어 있어도 → Accounting 탭으로 자동 전환.
- HTMX hx-get으로 리스트 교체

**검색 상태 바:**
- 현재 적용된 검색 조건을 한 줄로 표시
- 검색 없을 때: 숨김 또는 "전체 후보자 N명"
- 클릭하면 챗봇 모달 열림 (조건 수정 유도)

**후보자 카드:**
- 핵심 정보만: 이름, 현재 회사, 총 경력 연수, 학교, 상태 태그
- 클릭 → State 3 (상세 페이지)로 이동
- 무한 스크롤 (20건씩, HTMX hx-trigger="revealed")

#### State 2: 챗봇 모달 (대화형 검색)

```
┌───────────────────────────────┐
│  synco AI 검색          [—]   │  ← 헤더 (최소화 버튼)
├───────────────────────────────┤
│                               │
│  🤖 어떤 후보자를 찾으시나요?  │  ← AI 메시지
│                               │
│           회계 경력 있는 사람   │  ← 사용자 메시지
│                  찾아줘 🎤    │     (🎤 = 음성 입력 표시)
│                               │
│  🤖 회계 경력 후보자            │
│     30명을 찾았습니다 ✅       │
│                               │
│     10년차 이상이면서          │
│         대기업 경험 있는       │
│              사람만 🎤        │
│                               │
│  🤖 8명으로 좁혔습니다 ✅      │
│     리스트를 확인해보세요.     │
│                               │
├───────────────────────────────┤
│  [🎤] 텍스트로 입력하세요...   │  ← 입력 바
└───────────────────────────────┘
```

**모달 동작:**
- 플로팅 버튼 클릭 → 모달 열림 (오버레이 + 모달)
- 최소화(—) 또는 오버레이 클릭 → 모달 닫힘, 결과 리스트 업데이트됨
- 다시 열면 이전 대화 그대로 유지
- 검색 완료 시 "✅ N명을 찾았습니다" + "리스트를 확인해보세요" 메시지

**음성 입력 플로우:**
1. 마이크 버튼 클릭 → MediaRecorder로 녹음 시작
2. 녹음 중: 마이크 버튼 빨간색 + pulse 애니메이션
3. 말하기 끝 → 텍스트 변환 결과를 사용자 메시지로 표시
4. LLM이 텍스트를 분석 → 검색 쿼리 생성 → DB 검색 → 결과 반환
5. AI 메시지로 결과 개수 표시
6. 메인 화면 결과 리스트 업데이트 (HTMX swap)

**텍스트 입력:**
- 마이크 옆 텍스트 필드. Enter로 전송.
- 음성과 완전히 동등한 입력 수단.

**Multi-turn 동작:**
- 서버에 SearchSession 모델: session_id, 대화 히스토리, 현재 필터 조건
- 각 턴마다 LLM이 이전 조건 + 새 요청을 합쳐서 쿼리 생성
- "처음부터 다시" 또는 "초기화" → 새 세션 시작
- 세션은 서버에서 관리, 브라우저 sessionStorage에 session_id만 저장

## Technical Architecture

### 음성→검색 파이프라인

```
[마이크] → [Whisper API] → [텍스트] → [LLM: 자연어→검색조건] → [DB 쿼리] → [결과]
                                           ↑
                                    [SearchSession: 이전 조건들]
```

### 핵심 컴포넌트

**1. 음성 캡처 (프론트엔드)**
- `MediaRecorder` API로 오디오 녹음
- 녹음 완료 → 서버로 POST (audio/webm)
- 서버에서 Whisper API 호출 → 텍스트 반환

**2. 자연어→검색 변환 (서버)**
- `search_query_to_filters(session, user_text)` 헬퍼가 세션의 현재 필터 JSON + 새 쿼리 텍스트로 프롬프트를 구성, `call_llm_json()`에 전달
- LLM이 구조화된 검색 조건 JSON 반환:
  ```json
  {
    "filters": {
      "category": "Accounting",
      "min_experience_years": 10,
      "companies_include": ["현대", "삼성", "LG", "SK"],
      "education_level": null
    },
    "semantic_query": "대기업 회계 경력 10년 이상",
    "action": "narrow"  // "new" | "narrow" | "broaden"
  }
  ```
- `action: narrow` → 이전 결과에서 추가 필터
- `action: new` → 새 검색 (이전 조건 초기화)
- `action: broaden` → LLM이 완전한 대체 필터 JSON을 반환 (narrow와 같은 스키마). 사용자가 명시적으로 설정한 조건은 보존하고, LLM이 추론한 조건만 완화. 서버는 반환된 필터를 그대로 사용.

**3. 하이브리드 검색 (서버)**
- **구조화 필터:** ORM filter (category, experience years, company name)
- **시맨틱 검색:** pgvector cosine similarity (CandidateEmbedding)
- 두 결과를 결합: 구조화 필터로 좁힌 후 시맨틱 유사도로 정렬
- **기본 정렬:** 검색 없을 때(초기 화면/탭 클릭만) → 최근 업데이트순. 검색 시 → 시맨틱 유사도순.

**4. 세션 관리**
- `SearchSession` 모델: user, created_at, is_active
- `SearchTurn` 모델: session FK, user_text, ai_response, filters_json, result_count
- 브라우저 sessionStorage에 session_id 저장
- 모달 열 때 해당 세션의 대화 히스토리 로드
- **세션 라이프사이클:** 24시간 비활동 시 자동 만료 (is_active=False). 유저당 1개 활성 세션 (새 세션 생성 시 이전 비활성화). 브라우저 닫으면 sessionStorage 소멸 → 재방문 시 새 세션.

### 새 데이터 모델 (Phase 2 추가분)

```python
class SearchSession(BaseModel):
```

### Whisper API 통합

```python
# 비용 추정
# Whisper API: $0.006/분
# 평균 쿼리: 5-10초 → ~$0.001/쿼리
# 하루 100쿼리 → ~$0.10/일 → ~$3/월
# 비용 무시 가능

def transcribe_audio(audio_file) -> str:
    """Whisper API로 음성→텍스트 변환"""
    # openai 패키지 사용 (이미 pyproject.toml에 있음)
    client = openai.OpenAI()
    transcript = client.audio.transcriptions.create(
        model="whisper-1",
        file=audio_file,
        language="ko"
    )
    return transcript.text
```

## Risk Mitigation

### 음성 인식 정확도
- **Day 1 스파이크:** 실제 헤드헌터 쿼리 20개로 Whisper API 테스트.
- 회사명/학교명/직무명 혼합 한국어 쿼리 정확도 측정.
- 80% 미만이면 → LLM 후처리로 교정 (오타/오인식 수정 단계 추가).
- 최악의 경우: 텍스트 입력이 주 인터페이스가 되고 음성은 보조.

### LLM 자연어→검색 변환 정확도
- "10년차 이상 회계" 같은 쿼리가 정확한 DB 필터로 변환되는지.
- 프롬프트 튜닝: 10-20개 예시 쿼리로 few-shot learning.
- 변환 실패 → 전체 텍스트로 시맨틱 검색 fallback.

### Multi-turn 맥락 유지
- 대화가 5턴 이상 길어지면 LLM 컨텍스트 윈도우 부담.
- 해결: 대화 히스토리 대신 "현재 필터 조건 JSON"만 전달. 히스토리는 UI 표시용만.
- 결과: LLM에 보내는 건 항상 (현재 필터 + 새 쿼리)뿐. 간결.

## Resolved Decisions

1. **초기 화면:** 전체 후보자를 카테고리별로 표시 + 검색 상태 바에 "전체 후보자 N명". 첫 방문 시 챗봇 자동 열림.
2. **카테고리 탭과 음성 검색의 관계:** 음성이 주 인텐트. LLM이 카테고리를 추출하면 탭이 자동 전환. 추출 못하면 현재 탭 유지.
3. **인증:** 카카오 로그인 유지. 기존 OAuth 그대로 사용.
4. **반응형:** 모바일 우선. 챗봇은 모바일에서 하단 시트, 데스크탑에서 우하단 모달.
5. **LLM 프로바이더:** 검색 변환에 API 프로바이더 (Kimi/OpenRouter). claude_cli는 심층 분석용.
6. **쿼리 임베딩:** Gemini 동일 모델 (common/embedding.py 재사용).
7. **필터 검증:** pydantic 검증. 카테고리 미일치 시 null → 전체 검색.
8. **multi-tab:** 유저당 세션 수 제한 없음. 탭별 독립 세션.
