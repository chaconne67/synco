## Problem Statement

20년차 헤드헌터가 20만 건의 후보자 이력서를 Word/Excel 파일로 보관하고 있다. 채용건이 들어오면 폴더를 탐색하고 기억에 의존해서 후보자를 찾는다. 5명의 계약직 직원이 업무를 보조하지만 역량 편차가 커서 서류 품질이 들쭉날쭉하고, 파일 기반이라 데이터 공유/접근 제어도 어렵다.

한국에 활성 서치펌이 ~1,000곳 있고, 대부분 비슷한 상황이다. 글로벌 ATS(Bullhorn, Manatal 등)가 있지만 한국어 이력서 포맷과 중소 서치펌 업무 방식에 맞지 않아 채택률이 낮다.

## Demand Evidence

- 20년차 헤드헌터가 직접 의뢰. "도와달라"고 먼저 요청.
- 업체당 1,000만원 지불 의향 언급 (고객 본인의 시장 판단).
- 국내 활성 서치펌 ~1,000곳이 잠재 고객.
- 데이터 규모: 샘플 2,000건 제공, 실제 보유 20만 건.

## Status Quo

- 직무별 폴더 (Accounting, Sales, Engineering... 20개)에 Word 파일로 저장
- 파일명에 메타데이터 인코딩: `강솔찬.85.나디아.현대엠시트.인천대.doc`
- 서칭 = 폴더 탐색 + 머릿속 기억. 파일을 열기 전까지 상세 내용 모름.
- 5명 계약직이 인터뷰, 서류 작성 보조. 산출물 품질 편차 큼.
- 공유/접근 제어 불가. 파일 기반이라 권한 관리 자체가 안 됨.
- 글로벌 도구 미사용. 이유 추정: 가격, 한국어 비지원, 업무 방식 불일치.

## Target User & Narrowest Wedge

**Target User:** 한국 중소 서치펌 대표/시니어 리크루터. 10년+ 경력. 수천~수만 건의 후보자 파일을 직접 보유. Word/Excel로 관리 중. AI에 관심 있지만 기존 도구가 안 맞아서 못 쓰고 있는 사람.

**Narrowest Wedge:** Word/docx 이력서 파일을 올리면 자동으로 구조화된 DB로 전환하고, 이름/회사/직무/경력 등으로 검색할 수 있게 해주는 것. 이것만 되면 당장 쓴다.

## Constraints

- 기존 synco 코드베이스 활용 (Django 5.2 + HTMX + PostgreSQL + pgvector)
- Google Drive 연동 완료 (OAuth + 파일 다운로드 가능)
- 한국어 이력서 파싱이 핵심. 표준 양식 없음, 최소 5-6가지 포맷.
- LLM 기반 구조화 추출 필요 (규칙 기반으로는 불가)
- 20만 건 처리 시 비용/시간 계산 필요

## Premises

1. 한국 중소 서치펌의 핵심 자산은 자체 보유 후보자 DB이며, 이걸 구조화하는 도구가 실질적으로 없다.
2. 글로벌 ATS가 있지만, 한국어 이력서 포맷과 중소 서치펌 업무 방식에 맞지 않아 실제 채택률이 낮다.
3. 첫 고객(친구)의 20만 건 DB 구조화가 MVP이며, 이걸 성공적으로 해내면 다른 서치펌에도 팔 수 있다.
4. 단기 수익은 용역(업체당 ~1,000만원), 중기는 SaaS 월 구독료, 장기는 통합 DB의 네트워크 효과로 사업 가치가 급상승한다.

## EUREKA

글로벌 ATS들은 "채용 기업"을 위한 도구다. 서치펌이 쓸 수는 있지만 핵심 가치가 다르다. 서치펌에게 가장 중요한 건 자체 보유 후보자 DB. 20만 건의 Word 파일이 곧 20년간 축적한 사업 자산인데, 이걸 구조화하는 도구가 없다. synco는 "채용 기업의 ATS"가 아니라 "서치펌의 자산 관리 도구"다.

## Approaches Considered

### Approach A: DB-First (Selected)
- **Summary:** 이력서 파싱 + 검색에 집중. LLM으로 Word 파일 → 구조화 DB. 검색/필터링 UI. 친구분이 당장 쓸 수 있는 것에 집중.
- **Effort:** M (CC: ~2주)
- **Risk:** Low
- **Pros:** 빠른 가치 전달. 즉시 사용 가능. 이후 AI 매칭 추가 용이.
- **Cons:** AI 매칭은 나중. 초기 WOW 팩터 낮음.
- **Reuses:** synco Django 인프라, Google Drive 연동, pgvector

### Approach B: AI-Matching-First
- **Summary:** DB화와 AI 매칭을 동시 구축. JD 넣으면 후보자 추천까지.
- **Effort:** L (CC: ~4주)
- **Risk:** Med — 기본 검색도 늦어지는 리스크.
- **Pros:** 완성되면 WOW 팩터 큼.
- **Cons:** 4주 걸림. 친구분이 그동안 쓸 게 없음.

### Approach C: Platform-First
- **Summary:** 멀티테넌시 + 권한 관리까지 포함. 5명 직원이 각자 로그인.
- **Effort:** XL (CC: ~6주+)
- **Risk:** High — 과잉엔지니어링 리스크.
- **Pros:** 5명 직원 즉시 사용 가능.
- **Cons:** 6주+. 핵심 가치(검색) 전달이 늦어짐.

## Recommended Approach

**Approach A: DB-First.**

이유: 서치펌 대표가 가장 급하게 원하는 건 "찾는 것". 2주 안에 이력서를 올리면 검색되는 도구를 주면, 즉시 피드백이 온다. 거기서 나오는 피드백이 다음 단계(AI 매칭, 멀티유저)의 방향을 잡아준다.

20만 건 전체를 한번에 하는 게 아니라, 2,000건 샘플로 파이프라인 검증 → 20만 건 확장 순서.

## Implementation Plan (Eng Review 반영)

### Phase 1: 이력서 파싱 파이프라인 + 검증 시스템 (~2주)

#### 1. 새 candidates Django 앱 + DB 스키마
- Candidate, Category, Resume, Education, Career, Certification, LanguageSkill 모델
- ExtractionLog 모델 (추출 품질 추적 + 사람 수정 내역)
- Candidate 추가 필드: validation_status, raw_extracted_json, confidence_score, field_confidences
- Resume 추가 필드: processing_status (pending/downloaded/extracted/parsed/failed)
- Category fixture 20개 (Accounting, EHS, Engineer, ...)
- 기존 Contact/Interaction 모델 유지 (점진적 전환)

#### 2. Google Drive Sync 서비스
- 20개 카테고리 폴더 자동 인식
- Resume 모델에 drive_file_id 저장 (중복 방지)
- OAuth 토큰 자동 갱신 (장시간 배치)

#### 3. 텍스트 추출 서비스
- .docx → python-docx → raw text
- .doc → antiword → raw text (실패 시 LibreOffice headless fallback)
- Dockerfile에 antiword + LibreOffice 설치 (완료)

#### 4. 파일명 사전 그룹핑
- 파일명에서 이름+생년 정규식 추출
- 그룹당 최신 파일만 LLM 추출 (비용 절감)
- 매치 실패 → 개별 Candidate 생성 (graceful degradation)

#### 5. LLM 구조화 추출
- common/llm.py의 call_llm_json() 사용 (max_tokens=4000)
- concurrent.futures.ThreadPoolExecutor(max_workers=5) 병렬 처리
- JSON 파싱 실패 → 2회 재시도 → 3회 실패 시 status='failed'
- LLM에게 필드별 confidence score 반환 요청

#### 6. 3단계 검증 시스템
- **Layer 1:** LLM 자체 confidence (필드별 0.0-1.0)
- **Layer 2:** 규칙 기반 검증 (birth_year 범위, career 날짜 순서, 필수 필드)
- **Layer 3:** 파일명 vs LLM 추출 교차 검증
- 분류: HIGH(≥0.85) → auto_confirmed / MEDIUM(0.6-0.85) → needs_review / LOW(<0.6) → failed

#### 7. 리뷰 UI (간단한 HTMX 페이지)
- needs_review 후보자 리스트
- 원본 텍스트 + LLM 추출 결과 나란히 표시
- 확인/수정/거부 버튼
- 수정 내역 → ExtractionLog에 기록

#### 8. Management Command
- `python manage.py import_resumes --folder=Accounting --limit=100 --dry-run`
- 진행 상황 로깅, 완료 시 요약 리포트
- 재시작 시 status!='parsed'인 것만 처리

#### 9. 테스트
- 모델 테스트 (Candidate, Resume, Category 생성/유효성/관계)
- 텍스트 추출 테스트 (.doc/.docx 실제 샘플)
- 파일명 파서 테스트 (다양한 패턴)
- LLM 추출 테스트 (LLM mock + JSON 파싱)
- 파이프라인 idempotency 테스트
- LLM 품질 eval: 50건 랜덤 샘플 수동 검증

### Phase 2: 보이스 채팅 검색 UI (~1주)

**핵심 인터페이스 = 음성 자연어 검색.**

1. **카테고리 탭** — 20개 폴더 구조 그대로 탭으로 표시
2. **마이크 버튼** — Web Speech API로 음성 인식 → 텍스트 변환
3. **자연어 → 검색** — LLM이 자연어를 DB 쿼리로 변환 ("5년차 이상 회계 경력" → ORM filter)
4. **무한 스크롤 리스트** — 핵심 정보만 (이름, 학교, 회사, 경력 연수)
5. **후보자 상세 페이지** — 구조화된 프로필 + 원본 이력서
6. **CRUD** — 후보자 정보 추가/수정/삭제
7. **CandidateEmbedding** — pgvector 시맨틱 검색 (자연어 매칭)

### Phase 3: 확장 (후속)

- AI 매칭 (JD → 후보자 추천)
- 추천서 자동 생성
- 멀티유저 + 권한 관리
- 20만 건 대량 처리 파이프라인
- 채용 파이프라인 (JobOpening → CandidateMatch → Placement)

## Risk Mitigation

### LLM 파싱 실패 처리
- 파싱 실패/저품질 결과 → `needs_review` 플래그. 수동 보정 큐.
- 필드별 confidence score. 0.7 미만 → 리뷰 대상.
- LLM API 다운타임 → 재시도 로직 (3회, exponential backoff).

### .doc 파일 한국어 인코딩 리스크
- antiword가 EUC-KR/CP949 테이블 처리 실패 가능.
- Phase 1 Day 1 스파이크: 20개 샘플 .doc 파일로 antiword 테스트.
- 실패 시 fallback: LibreOffice headless → PDF → 텍스트 추출.
- .hwp 파일 존재 여부 친구분에게 확인 필요.

### LLM 비용 추정 (Phase 1 필수 산출물)
- 100건 처리 후 평균 토큰 수 측정 → 파일당 비용 계산.
- 예상: Claude Haiku $0.01/건 → 20만 건 = ~$2,000.
- GPT-4o-mini와 비교 벤치마크.

### 프롬프트 엔지니어링 시간
- 5-6가지 이력서 포맷에 90%+ 정확도 달성에 2-3일 소요 예상.
- Phase 1에서 프롬프트 반복 튜닝 시간 명시적 할당.

### 중복 판별 키
- 1차: 이름 + 생년 (파일명 + LLM 추출 교차)
- 2차: 이름 + 생년 + 첫 번째 경력 회사
- 생년 누락 시: 이름 + 전화번호/이메일. 없으면 separate Candidate로 유지.

### 디스크 공간 (200K 확장 시)
- 개발 서버 13GB 여유. 200K 파일(10-20GB)은 저장 불가.
- DB 서버 /mnt에 73GB 여유. 200K 확장 시 임시 파일 DB 서버에 저장 또는 스트리밍 처리.

2. **첫 고객 피드백 루프:** 2,000건 파싱 후 친구분의 피드백 → Phase 2 방향 조정.

6. **음성 인식 정확도:** 한국어 Web Speech API의 전문용어(직무명, 회사명) 인식 정확도.

## Success Criteria

- [ ] 2,000건 샘플 이력서 → 구조화 DB 전환 완료
- [ ] 파싱 정확도 90%+ (50건 랜덤 샘플링, 수동 검증. 정확도 = 올바른 필드 / 전체 필드)
- [ ] 검증 시스템이 추출 오류 80%+ 자동 감지
- [ ] 이름/회사/직무로 검색 → 3초 이내 결과
- [ ] 파이프라인 중단 후 재시작 → 이어서 처리
- [ ] 친구분이 "이전보다 낫다"고 판단 (정성 피드백)
- [ ] 20만 건 확장: 파일당 비용 산출, 배치 아키텍처 문서화, 예상 총 처리 시간 확정

## What I noticed about how you think

- "첫 번째 당장 현금 할 수 있는 것은..." — 바로 수익화 경로를 먼저 생각했다. 대부분의 빌더는 기능부터 생각하는데, 당신은 비즈니스 모델부터 봤다.
- "한 업체당 천만원 정도를 받을 수 있을 것이다" — 이건 고객(친구분)이 직접 한 시장 판단이다. 빌더 혼자 추측한 게 아니라 20년 업계 경험에서 나온 가격이다. 이게 가장 강력한 수요 증거.
- "데이터를 공유해서 전체 DB를 한군데 몰아넣으면 굉장한 사업 기회" — 개별 용역 너머의 플랫폼 비전을 이미 보고 있다. 데이터 네트워크 효과를 직관적으로 이해하고 있다.
- 기존 synco 플랫폼을 버리지 않고 피보팅하겠다는 판단. 기술 스택을 재활용하는 실용성.
