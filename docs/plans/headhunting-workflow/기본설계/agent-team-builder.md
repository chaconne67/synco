# Agent Team Builder

프로그램 및 워크플로우를 다각도로 점검하는 에이전트 팀을 구성하는 스킬.
각 에이전트는 서브에이전트가 아닌 **독립적인 에이전트**로, 모든 도구와 스킬 접근 권한을 가진다.

**전제:** `/prompt-golden-rules` 스킬이 설치되어 있어야 한다 (프롬프트 검사 시 사용).

---

## 발동 조건

**트리거:** 사용자가 "멀티 에이전트 팀을 구성해서 검사해줘" 유형의 요청을 할 때.
예: "멀티 에이전트로 점검해줘", "에이전트 팀 구성해서 감사해줘", "팀으로 검사해줘"

**발동하지 않는 경우:**
- 단일 파일/함수 수준의 변경 → 서브에이전트 하나로 충분
- 범위가 좁혀진 버그 수정 → `/investigate`
- PR 리뷰 → `/review`
- 특정 도메인만 해당 → 해당 도메인 에이전트 하나로 충분

## 핵심 원칙

### 목적이 먼저다

프로그램의 목적을 모르면 코드를 봐도 맞는지 틀린지 판단할 수 없다.
**반드시 목적 파악이 선행**되어 나머지 에이전트들의 통과 기준을 정의해야 한다.

목적 파악은 오케스트레이터가 팀 구성 전에 직접 수행한다. 다음을 입력으로 사용한다:
- 대상 코드 자체 (파일, 모듈, 파이프라인)
- 코드가 속한 앱의 README, 문서, CLAUDE.md 등 프로젝트 맥락
- 코드의 호출자와 소비자 — 이 코드의 출력을 누가, 어떻게 쓰는가

이를 바탕으로 두 수준의 목표를 정의한다:
- **추상적 목표** — 프로그램 전체가 달성해야 할 가치. 전체 구조의 통과 기준이 된다.
- **구체적 목표** — 하위 메서드/코드 블록이 산출해야 할 결과물. 개별 코드의 pass/fail 기준이 된다.

목적에는 기능적 목표뿐 아니라 **데이터 무결성, 보안, 안전성 같은 비기능 요구도 포함**된다. 이를 목적 파악 단계에서 누락하면 검사 에이전트가 비기능 이슈를 "목적 외"로 취급하여 놓칠 수 있다.

통과 기준은 다음 조건을 만족해야 한다:
- **검증 가능** — 에이전트가 도구를 써서 실제로 확인할 수 있는 형태
- **목적 연결** — 왜 이 기준이 필요한지 프로그램의 목적으로 설명 가능
- **판정 가능** — 통과/미통과를 판단할 수 있을 만큼 구체적

> 나쁜 기준: "이력서 파싱이 정확해야 한다"
> 좋은 기준: "이력서에서 추출한 경력 기간이 원본 PDF의 텍스트와 일치해야 한다 — 이 데이터가 후보자 매칭의 입력이 되기 때문"

### 팀 구성은 프로그램의 성격이 결정한다

아래는 점검 관점의 **예시**이지, 고정 목록이 아니다.
오케스트레이터가 프로그램의 성격을 분석한 뒤, 어떤 관점의 검사가 필요한지 판단하여 팀을 구성한다.

| 관점 | 언제 필요한가 | 참고 |
|------|-------------|------|
| 코드 검사 | 거의 항상 | 데이터 흐름, 문법, 인풋/아웃풋 규격, 구조 |
| 템플릿 정합성 | 프론트엔드 템플릿이 있을 때 | 백엔드 데이터와 템플릿 간 불일치 |
| UI/UX | 사용자에게 보이는 화면이 있을 때 | 디자인 시스템, 시각적 품질. `/browse` 필수 |
| 인터랙션 | 사용자 조작이 있을 때 | 실제 동작 여부, 기대 부합, 전환 자연스러움. `/browse` 필수 |
| 프롬프트 | LLM을 호출하는 하이브리드 프로그램일 때 | 아래 별도 설명 |
| 보안, 성능, 접근성 등 | 프로그램 성격에 따라 | 오케스트레이터가 필요성을 판단 |

**의심스러우면 만들지 않는다.** 과잉 생성과 부족 생성의 비용은 비대칭이다. 관점이 부족하면 나중에 에이전트를 추가할 수 있지만, 과잉 생성된 에이전트는 근거 없는 발견 사항을 만들어내고 오케스트레이터의 종합 판단을 흐린다. 에이전트를 추가하는 비용보다 노이즈를 걸러내는 비용이 더 크다.

### 실패 비용의 비대칭

검사에서 **놓침(거짓 음성)은 과잉 보고(거짓 양성)보다 비싸다.** 과잉 보고는 오케스트레이터가 걸러낼 수 있지만, 놓친 문제는 복구할 기회 자체가 없다. 따라서 에이전트는 의심이 들면 보고하되, 근거와 함께 확신도를 명시하여 오케스트레이터가 판단할 수 있게 한다.

### 자율성의 원칙

이 스킬은 **검사의 관점과 원칙**을 제공하지, 구체적인 체크리스트를 지정하지 않는다.
LLM은 체크리스트를 받으면 기계적으로 따라가고, 원칙을 받으면 프로그램의 특성에 맞는 최적의 검사를 스스로 설계한다. 체크리스트는 작성자가 예상한 케이스만 잡지만, 원칙을 이해한 에이전트는 예상하지 못한 문제도 발견할 수 있다.

## 워크플로우

```
[0] 오케스트레이터: 목적 파악 + 사전 질문
        │
        ▼  모호한 부분이 있으면 사용자에게 질문하여 확정
        │
[1] 오케스트레이터: 통과 기준 수립 + 팀 구성
        │
        ▼  TeamCreate로 검사 팀 생성, 통과 기준 전달
        │
[2] 검사 에이전트들 (Teammates, 병렬 실행)
        │
        ▼
   오케스트레이터가 결과 종합 → 최종 보고
```

### 최종 산출물

#### 보고서 저장 구조

모든 점검 결과는 반드시 **문서로 보고**한다. 보고서는 다음 경로에 저장한다:

```
docs/inspection/multi-agent/
└── {YYYY-MM-DD}-{작업요약}/          ← 점검 실행 시 생성
    ├── 00-orchestrator-report.md     ← 오케스트레이터 종합 보고 (최종)
    ├── {관점}-inspection.md          ← 각 검사 에이전트의 개별 리포트
    └── ...
```

- **폴더 이름**: `날짜-작업내용` 형식. 예: `2026-04-05-extraction-pipeline`, `2026-04-05-candidate-list-page`
- **개별 리포트**: 각 검사 에이전트가 자신의 발견 사항을 파일로 저장. 파일명은 관점을 반영 (예: `code-inspection.md`, `prompt-inspection.md`, `template-inspection.md`)
- **종합 보고**: 오케스트레이터가 모든 개별 리포트를 대조한 뒤 `00-orchestrator-report.md`에 최종 판정을 작성

#### 보고 내용

오케스트레이터의 종합 보고는 다음을 포함한다:
- **종합 판정** — 프로그램이 목적을 달성하는지 여부 (통과/미통과/조건부 통과)
- **발견 사항 목록** — 각 항목에 심각도(차단/주의/참고), 근거(코드 위치, 재현 조건), 수정 방향을 포함
- **교차 이슈** — 개별 에이전트 결과를 대조하여 발견한 문제를 별도 섹션으로 분리

이 보고는 **발견과 판단**까지만 수행한다. 수정은 사용자의 지시가 있을 때 별도로 진행한다.

### 워크플로우 상세

0. **사전 질문** — 오케스트레이터가 코드와 문서를 읽고 목적을 파악한 뒤, 코드만으로는 확정할 수 없는 부분이 있으면 **팀 구성 전에** 사용자에게 질문한다. 예: 비즈니스 맥락, 의도된 동작과 버그의 경계, 검사 범위의 우선순위. 모호한 전제 위에 팀을 구성하면 에이전트들이 잘못된 기준으로 검사하게 된다.
1. **통과 기준 수립 + 팀 구성** — 사전 질문으로 확정된 맥락을 바탕으로 통과 기준을 수립하고, 필요한 검사 관점을 결정하여 TeamCreate로 팀을 생성한다.
2. **검사 에이전트들** — 통과 기준을 입력으로 받아 병렬 실행. 각 에이전트는 발견 사항을 정규화된 형식(파일/라인, 재현 조건, 1-2문장 판단 근거, 필요 시 최소 코드 인용)으로 오케스트레이터에 전달한다.
3. **오케스트레이터** — 개별 에이전트가 볼 수 없는 **교차 이슈**를 발견하고 종합 판정을 내린다. 교차 이슈란 한 관점에서는 정상이지만 다른 관점과 대조하면 드러나는 문제이다:
   - 코드 검사는 통과했지만 프롬프트 검사에서 LLM 응답 규격이 코드의 파싱 로직과 불일치하는 경우
   - 템플릿 정합성은 통과했지만 코드 검사에서 특정 조건의 데이터가 null로 넘어가 템플릿이 깨지는 경우
   
   에이전트 간 결과가 충돌하면 프로그램의 목적에 더 가까운 쪽을 우선한다.

## 프롬프트 검사에 대하여

스크립트와 언어 모델을 함께 사용하는 하이브리드 프로그램에서, 프롬프트는 코드만큼 중요하다.
프롬프트가 부정확하면 나머지 코드가 아무리 잘 짜여 있어도 프로그램 전체가 목적을 달성하지 못한다.

프롬프트 검사는 두 층위로 이루어진다:

**기본층: `/prompt-golden-rules` 스킬 적용**
프롬프트 설계의 기본 원칙(하드코딩 금지, 충분한 맥락, 최적화 등) 준수 여부를 점검한다.

**심화층: 목적 기반 정제**
Golden Rules를 통과한 프롬프트라도, 프로그램의 목적과 통과 기준에 비추어 충분히 구체적이고 정확한지 검사한다.
- 이 프롬프트가 프로그램의 목적이 원하는 결과를 최대한 정확하게 이끌어내는가
- LLM에게 기대하는 결과물이 해당 코드 블록의 통과 기준에 부합하는가
- LLM 응답의 규격과 후속 코드의 기대가 일치하는가

> 통과 예: 이력서 파싱 프롬프트가 "경력 기간을 시작일-종료일 쌍으로 추출하라"고 지시하고, 후속 코드가 동일한 쌍 구조를 기대 — 목적(경력 기간 비교)과 규격이 일관됨
> 미통과 예: 프롬프트는 "경력을 요약하라"고 지시하지만, 후속 코드는 개별 시작일/종료일을 파싱 — 프롬프트의 출력이 코드의 입력 기대와 불일치

## 시나리오 예시

같은 프로젝트 안에서도 대상에 따라 팀 구성이 달라진다.

**시나리오 A: 이력서 파싱 파이프라인** (Django + LLM 하이브리드)
→ 오케스트레이터가 코드 검사, 템플릿 정합성, 프롬프트 검사 3개 관점을 선택.
UI/UX·인터랙션은 화면이 없으므로 생략.

**시나리오 B: 후보자 목록 페이지** (Django + HTMX, LLM 없음)
→ 오케스트레이터가 코드 검사, 템플릿 정합성, UI/UX 3개 관점을 선택.
프롬프트 검사는 LLM이 없으므로 생략.
