# 스킬 설계서: plan-verification (플랜 구현 점검 스킬)

## 1. 스킬 개요

| 항목 | 내용 |
|------|------|
| **이름** | `plan-verification` |
| **유형** | Technique (단계별 절차가 있는 실행 스킬) |
| **트리거** | 구현 플랜이 존재하고 코드가 작성된 후, 플랜 대비 구현을 점검할 때 |
| **출력** | 전 항목 PASS된 최종 점검 보고서 |

```yaml
name: plan-verification
description: >
  Use when an implementation plan exists and code has been written,
  and you need to verify that the implementation matches the plan.
  Triggers: "플랜대로 됐는지 확인", "점검해줘", "리뷰하고 수정해줘",
  or referencing a plan file path with a verification request.
```

---

## 2. 이 스킬이 필요한 이유

### 문제

구현 플랜대로 코드를 작성한 뒤 "다 했다"고 보고하는 것은 위험하다.
플랜과 코드 사이에는 항상 갭이 존재한다:

| 갭 유형 | 예시 | 놓쳤을 때 비용 |
|---------|------|---------------|
| 누락 | 플랜의 한 Step을 빼먹음 | 기능 미동작, 운영 장애 |
| 변형 | 상수값/열거형이 플랜과 다름 | 비즈니스 로직 오류 |
| 프롬프트 품질 | LLM 시스템 프롬프트에 하드코딩 규칙 | LLM이 엣지 케이스에서 실패 |
| 테스트 부재 | 플랜 명시 테스트 중 일부만 작성 | 회귀 버그 미탐지 |
| 설계 원칙 위반 | OR 필터여야 하는데 AND 필터 | 쿼리 결과 0건 |

단일 리뷰어가 이 모든 갭을 잡기 어렵다. 관점이 다른 여러 에이전트가 병렬로
점검하고, 수정 후 재검증하는 루프가 필요하다.

**이 스킬의 실패 비용:** false negative(문제를 놓침)가 false positive(멀쩡한 것을 지적)보다 비싸다. 놓친 불일치는 운영에서 장애로 나타나지만, 잘못된 지적은 수정 단계에서 "이건 아니다"로 걸러진다. 따라서 모든 에이전트는 의심스러우면 FAIL로 보고한다.

### 기존 스킬과의 관계

| 기존 스킬 | 역할 | 이 스킬과의 차이 |
|-----------|------|-----------------|
| `review` | PR 단위 코드 리뷰 | 플랜 문서와의 정합성은 보지 않음 |
| `investigate` | 버그 원인 추적 | 이미 동작하는 코드의 정합성 점검이 아님 |
| `prompt-golden-rules` | 프롬프트 품질 점검 | 단독 스킬. 이 스킬의 하위 단계로 통합 |
| `agent-team-builder` | 멀티 에이전트 팀 구성 | 범용 프레임워크. 이 스킬은 "플랜 점검" 특화 |

이 스킬은 위 스킬들을 **조합**하여 "플랜 → 구현 → 점검 → 수정 → 재검증" 루프를 제공한다.

> **스킬 변환 참고:** 이 섹션(배경, 기존 스킬 비교)은 설계 문서용이다. SKILL.md로 변환 시 "When to Use / When NOT to Use" 섹션으로 축약하고, 이 배경 설명은 포함하지 않는다.

## 3. 전체 워크플로우

```
┌─────────────────────────────────────────────────────────────┐
│  INPUT: 플랜 파일 경로                                        │
└──────────────────────┬──────────────────────────────────────┘
                       ▼
              ┌────────────────┐
              │ Phase 1: 분석   │  플랜 읽기 → 카테고리 도출 → 체크리스트 생성
              └───────┬────────┘
                      ▼
              ┌────────────────┐
              │ Phase 2: 점검   │  카테고리별 에이전트 병렬 디스패치
              └───────┬────────┘
                      ▼
              ┌────────────────┐
              │ Phase 3: 종합   │  보고서 병합 → FAIL/REVIEW 분류
              └───────┬────────┘
                      ▼
                 FAIL 있나? ──No──→ Phase 5: 최종 보고
                      │
                     Yes
                      ▼
              ┌────────────────┐
              │ Phase 4: 수정   │  오케스트레이터가 FAIL 수정 → 검증 실행
              └───────┬────────┘
                      ▼
                 Phase 2로 복귀 (전체 재검증)
```

### Phase 1: 분석 — 플랜에서 체크리스트 생성

오케스트레이터가 플랜 파일을 읽고, 플랜의 **변경 유형**에 따라 점검 카테고리를 도출한다.

**카테고리 도출 원칙:** 플랜에서 "어떤 파일에 무엇이 있어야 하는가"를 서술하는 모든 구절을 하나의 체크리스트 항목으로 변환한다. 플랜이 단계(Step)로 구조화되어 있으면 단계별로, 서술형이면 문장 단위로 추출한다.

일반적으로 나타나는 카테고리와 생성 조건:

| 카테고리 | 생성 조건 | 항목 추출 기준 |
|---------|----------|--------------|
| **코드 정합성** | 항상 (코드 변경이 있는 플랜이므로) | 플랜의 각 Step에서 파일, 위치, 기대 내용을 추출 |
| **테스트 완전성** | 플랜에 테스트 정의가 있을 때 | 테스트 함수명, 검증 시나리오를 추출 |
| **실행 검증** | 항상 | 플랜의 "검증 방법" 섹션에서 실행 커맨드를 추출. 섹션이 없으면 프로젝트 설정(CLAUDE.md, pyproject.toml 등)에서 테스트/린트 커맨드를 식별 |
| **설계 원칙 준수** | 플랜에 설계 원칙/알려진 한계 섹션이 있을 때 | 보장해야 할 동작과 코드에서 확인할 지점을 추출 |
| **LLM 프롬프트 품질** | 플랜의 수정 대상에 LLM 프롬프트가 포함될 때 | Phase 2에서 에이전트가 자체적으로 코드를 읽고 프롬프트를 식별. 이유: 프롬프트가 어디에 있는지는 플랜이 아닌 코드가 알려주므로, Phase 1에서 미리 추출하지 않는다 |

플랜에 위 표에 없는 변경 유형(DB 마이그레이션, API 스키마 등)이 있으면 해당 유형의 카테고리를 추가한다. 플랜에 해당 유형이 없으면 카테고리를 생성하지 않는다.

**정보 보존:** 체크리스트 생성 시 플랜 원문의 맥락(설계 이유, 제약 배경 등)이 소실될 수 있다. 각 에이전트에게 체크리스트와 함께 **플랜 원문 경로**를 전달하여, 판단이 필요할 때 원문을 참조할 수 있게 한다.

### Phase 2: 점검 — 멀티 에이전트 병렬 디스패치

Phase 1에서 도출된 카테고리 수에 따라 필요한 에이전트만 디스패치한다. 각 에이전트는 독립적으로 동작하며 오케스트레이터에게 구조화된 보고서를 반환한다.

에이전트 배정: 코드 정합성 → Code Reviewer, 테스트 완전성+실행 검증 → Test Verifier, LLM 프롬프트 품질 → Prompt Quality Reviewer, 설계 원칙 → Design Principle Verifier. 카테고리가 없으면 해당 에이전트는 생성하지 않는다.

#### 에이전트 1: Code Reviewer (코드 정합성)

**역할:** 플랜 문서와 실제 코드를 대조하여 의미적 불일치를 식별

**실패 비용:** false negative(불일치를 놓침)가 더 비싸다. 놓친 불일치는 운영에서 발견될 때까지 잠복한다. 의심스러우면 FAIL로 보고하라.

**입력:**
- 플랜 파일 경로 (체크리스트 + 원문 참조용)

**작업:**
1. 플랜 파일을 읽는다
2. 플랜에서 언급하는 모든 파일을 읽는다
3. 각 항목에 대해 플랜의 기대값과 실제 코드를 비교한다
4. 불일치 발견 시 `{ 항목코드, 파일, 위치, 기대값, 실제값, 심각도 }` 기록

**심각도 판단 기준:**
- critical: 기능 동작에 영향 (잘못된 로직, 누락된 필터, 틀린 상수값)
- minor: 기능에는 영향 없으나 플랜과 불일치 (코드 위치, 변수명, 주석)

**판단 경계 사례:**
- 플랜: `["admin", "editor", "viewer"]` / 코드: `["admin", "viewer", "editor"]` → **PASS** (순서는 의미적 차이 아님)
- 플랜: `["admin", "editor", "viewer"]` / 코드: `["admin", "editor"]` → **FAIL** (항목 누락은 기능 차이)
- 플랜: `sorted(set(...))` / 코드: `list(dict.fromkeys(...))` → **PASS** (구현 방식이 다르지만 결과 동일)

**출력 형식:**
```
## Code Review Report
| 항목 | 상태 | 심각도 | 상세 |
|------|------|--------|------|
| A1   | PASS | —      | — |
| A2   | FAIL | critical | 기대: X, 실제: Y |
```

#### 에이전트 2: Test Verifier (테스트 완전성 + 실행 검증)

**역할:** 테스트 완전성 확인 + 실제 테스트/린트 실행

**실패 비용:** false negative(누락된 테스트를 놓침)가 더 비싸다. 테스트가 없으면 회귀 버그를 잡을 수 없다. 의심스러우면 FAIL로 보고하라.

**입력:**
- 플랜 파일 경로 (테스트 정의 + 검증 방법 섹션)

**작업:**
1. 테스트 파일을 읽고 플랜의 테스트 목록과 대조한다
2. 각 테스트의 시나리오가 플랜 설명과 일치하는지 확인한다
3. 플랜의 "검증 방법" 섹션(또는 프로젝트 설정)에서 추출한 커맨드를 실행한다

**판단 경계 사례:**
- PASS: 플랜은 "4개 레코드 생성" 코드는 "3개 + 1개 제외 대상" — 검증 논리 동일
- FAIL: 플랜은 "입력값 변형 테스트(대소문자, 공백)" 요구, 코드에 해당 테스트 없음 — 커버리지 누락

**출력 형식:**
```
## Test Verification Report
### 테스트 완전성
| 항목 | 상태 | 상세 |
### 실행 검증
| 커맨드 | 상태 | 출력 요약 |
```

#### 에이전트 3: Prompt Quality Reviewer (LLM 프롬프트 품질)

**역할:** 코드 내 LLM 프롬프트를 `prompt-golden-rules` 스킬로 점검

**실패 비용:** false negative(프롬프트 결함을 놓침)가 더 비싸다. 프롬프트 결함은 LLM이 실전에서 실패할 때까지 발견되지 않는다. 의심스러우면 지적하라.

**입력:**
- 플랜의 수정 대상 파일 목록

**작업:**
1. 수정 대상 파일에서 LLM API 호출 함수(예: call_llm, generate 등)를 찾고, 그 인자로 전달되는 문자열(시스템 프롬프트, 스키마 정의, 규칙 목록)을 역추적하여 식별한다
2. `prompt-golden-rules` 스킬의 세 가지 원칙으로 점검한다:

   **하드코딩 점검:** 프롬프트 내 규칙이 열거/땜질/조건분기/임계값에 해당하는가. 하드코딩이면 "왜 LLM이 이 규칙 없이 틀렸을까" → 원인이 맥락 부족이면 맥락 보충이 정답. 출력 형식 사양(JSON 스키마 등)은 하드코딩이 아님.

   **맥락 충분성 점검:** 각 지시에 "출력 용도 / 이유(WHY) / 실패 비용 / 입력 특성" 중 빠진 것이 있는가. "~하지 마세요" 규칙에 이유가 없으면 FAIL.

   **사례 적정성 점검:** 사례가 해당/미해당 경계를 쌍으로 보여주는가. 원칙만으로 충분히 명확하면 사례 불필요.

   **균형 + 안티패턴:** 예외 규칙 3개 이상 누적 → 프레이밍 재설계 신호. 제거해도 출력이 달라지지 않는 문장 → 토큰 낭비. 한 프롬프트에 여러 역할(추출+판단+포맷팅) 혼합 → 분리 검토.

3. 문제 발견 시 원문 인용 + 위반 원칙 + Before/After 수정안 제시

4. **수정안에 사례를 추가할 때 — 웹검색으로 도메인 맥락 보강:**
   프롬프트에 사례가 부족하거나, 기존 사례가 경계를 제대로 보여주지 못할 때, 에이전트 자신의 추측이 아닌 **프로젝트 도메인의 실제 관행**을 근거로 사례를 작성한다.

   **왜 웹검색이 필요한가:** 프롬프트의 사례는 LLM이 판단 경계를 학습하는 데 쓰인다. 에이전트가 도메인 지식 없이 사례를 지어내면 실제 업계 관행과 어긋나서, LLM이 잘못된 경계를 학습한다. 예: 의료 도메인에서 "중증도 분류" 사례를 만들 때 KTAS(한국형 응급환자 분류도구) 5단계 체계를 모르면 잘못된 경계를 제시하고, 물류 도메인에서 "배송 상태 분류" 사례를 만들 때 업계 표준 상태값을 모르면 실제 운영과 맞지 않는다.

   **웹검색 실행 조건:**
   - 프롬프트가 도메인 특화 용어/분류를 다루고, 에이전트가 해당 도메인의 관행을 확신할 수 없을 때
   - 수정안에 추가할 사례가 도메인의 실제 사용 패턴을 반영해야 할 때
   - 프롬프트의 맥락(WHY) 보충에 업계/기술 배경 정보가 필요할 때

   **웹검색하지 않는 경우:**
   - 프로그래밍 구문, 프레임워크 API 등 에이전트가 이미 정확히 아는 영역
   - 사례가 도메인 독립적인 일반 패턴일 때 (예: JSON 스키마 형식)

   **검색 → 사례 작성 흐름:**
   1. 프롬프트의 도메인 키워드를 식별 (예: "KTAS 분류", "HS Code 체계", "ISCO 직업 분류")
   2. 해당 키워드로 웹검색하여 업계 관행/분류 체계/실사용 사례를 수집
   3. 수집된 정보를 바탕으로 해당/미해당 경계를 정확히 보여주는 사례 쌍을 작성
   4. 사례 출처가 검색 결과임을 수정안에 명기 (신뢰도 판단용)

   **판단 경계 사례:**
   - 검색 필요: 프롬프트가 업계 특화 분류 체계를 다루는데 에이전트가 해당 분류의 경계를 모름 (예: 의료 중증도 5단계, 물류 HS Code 6자리 vs 10자리) → 검색하여 실제 통용 범위 확인
   - 검색 불필요: 프롬프트의 규칙에 이유(WHY)가 빠짐 → 이유는 코드 로직에서 도출 가능, 도메인 지식이 아닌 구조적 문제이므로 검색 불필요

**판단 경계 사례:**
- PASS: "카테고리에 해당하면 category_tags에, 개별 항목이면 item_keywords에 넣으세요." — 판단 기준이 명확하고 원칙으로 일반화 가능
- FAIL: "category_tags와 item_keywords를 동시에 사용하지 마세요" — WHY 없음. "AND 결합으로 0건이 되기 때문"을 추가해야 함

**출력 형식:**
```
## Prompt Quality Report
| 원칙 | 상태 | 상세 |
|------|------|------|
| 하드코딩 | FAIL | 규칙 N. Before: "..." → After: "..." |
| 맥락 | PASS | — |
```

#### 에이전트 4: Design Principle Verifier (설계 원칙 준수)

**역할:** 설계 원칙이 코드와 테스트에서 실제로 보장되는지 확인

**실패 비용:** false negative(원칙 위반을 놓침)가 더 비싸다. 원칙 위반은 특정 입력에서만 드러나는 조용한 버그를 만든다. 의심스러우면 FAIL로 보고하라.

**입력:**
- 플랜 파일 경로 (설계 원칙 / 알려진 한계 섹션)

**작업:**
1. 각 설계 원칙에 대해 코드에서 해당 원칙이 보장되는 지점을 찾는다
2. 테스트에서 해당 원칙을 검증하는 테스트가 있는지 확인한다
3. 테스트에서 커버하지 못하는 엣지 케이스를 식별한다

**판단 경계 사례:**
- PASS: "대소문자 무시" 원칙에 대해 `test_case_insensitive_matching` 존재 — 테스트가 원칙을 직접 검증
- FAIL: "미인식 입력 폴백 처리" 원칙에 대해 해당 테스트 없음 — 원칙이 테스트로 보장되지 않음

**출력 형식:**
```
## Design Principle Report
| 원칙 | 상태 | 보장 지점 | 미커버 엣지 케이스 |
```

### Phase 3: 종합 — 보고서 병합

오케스트레이터가 에이전트 보고서들을 수집하고:
1. 전체 항목을 PASS/FAIL/REVIEW로 분류
2. REVIEW 판정: 에이전트가 FAIL로 보고했으나, 코드가 플랜과 다르되 테스트가 통과하고 설계 원칙도 충족하는 경우, 오케스트레이터가 FAIL을 REVIEW로 재분류한다. 이유: 플랜 자체의 오류일 수 있으므로, 자동 수정 대신 사용자 판단에 위임한다
3. FAIL이 0개이면 Phase 5로 이동 (REVIEW만 남은 경우도 Phase 5로 이동하여 사용자에게 보고)
4. FAIL이 있으면 심각도 순(critical → minor)으로 정렬하여 Phase 4로 이동. 이유: critical은 기능 영향이 있어 다른 수정의 전제 조건이 될 수 있다

### Phase 4: 수정 — FAIL 항목 코드 수정

**수정 주체:** 오케스트레이터가 직접 수정한다. 이유: 에이전트들의 보고서를 종합하여 충돌을 해결해야 하므로 전체 맥락을 가진 오케스트레이터가 수정해야 한다. 수정 후 재검증(Phase 2)은 에이전트를 새로 디스패치하므로 "자기 수정 자기 검증" 문제는 발생하지 않는다 — 검증 에이전트는 수정 이력을 모르고 코드만 본다.

**수정 규칙:**
- 플랜 문서가 기준이다. 이유: 플랜은 사용자가 의도적으로 설계한 사양이므로, 의도 없는 이탈은 버그다. REVIEW 판정 항목(Phase 3에서 재분류된 것)은 수정하지 않고 최종 보고에 포함한다.
- 프롬프트 품질 FAIL은 golden rules 수정안을 적용한다
- 수정 범위는 플랜에서 명시한 변경 대상 파일로 제한한다. 플랜이 금지하는 변경(예: DB 스키마 변경 금지)은 수정 시도하지 않는다. 이유: 이 스킬의 범위는 "기존 플랜의 구현 정합성 점검"이며, 플랜 범위 밖의 변경은 데이터 손실 등 별도 위험이 있다.
- 수정 후 플랜의 "검증 방법" 섹션(또는 프로젝트 설정)에서 추출한 테스트/린트 커맨드를 실행한다

수정 완료 후 Phase 2로 복귀하여 **전체 재검증**한다. 이유: 수정이 다른 항목에 부작용을 일으킬 수 있으므로, 수정된 항목만 재검증하면 부작용을 놓친다.

## 4. 에이전트 프롬프트 작성 원칙

각 에이전트에게 전달하는 프롬프트 자체도 골든룰을 따라야 한다.

### 왜 에이전트 프롬프트에 골든룰이 필요한가

에이전트 프롬프트는 "LLM에게 작업을 지시하는 프롬프트"다.
골든룰 위반이 있으면 에이전트가 엣지 케이스에서 잘못된 판단을 한다.

경계 사례 (나쁜 프롬프트 → 좋은 프롬프트):

| 나쁜 프롬프트 | 위반 원칙 | 좋은 프롬프트 |
|-------------|----------|-------------|
| "A1~A10을 순서대로 확인하세요" | 하드코딩: 항목이 바뀌면 대응 불가 | "첨부된 체크리스트의 각 항목에 대해, 플랜의 기대값과 코드의 실제값을 비교하라" |
| "코드를 확인하세요" | 맥락 부족: 기준과 이유 없음 | "플랜 문서가 기준이다. 플랜은 사용자가 승인한 의도이므로, 코드가 이 의도에서 벗어났는지 확인하라" |
| "불일치를 보고하세요" | 경계 불명확 | "의미적 차이가 있는 불일치를 보고하라. 순서/포맷팅 차이는 PASS. 항목 누락/로직 변경은 FAIL" |

### 에이전트 프롬프트 구성 규칙

| 요소 | 필수 여부 | 내용 |
|------|----------|------|
| **목적 (WHY)** | 필수 | 이 에이전트가 왜 존재하는가. 점검하지 않으면 어떤 문제가 발생하는가 |
| **입력 (WHAT)** | 필수 | 읽어야 할 파일 경로, 체크리스트 항목 (구체적) |
| **판단 기준 (HOW)** | 필수 | PASS/FAIL을 어떻게 판정하는가. 원칙으로 서술 (하드코딩 X) |
| **실패 비용** | 필수 | false negative vs false positive 중 어느 쪽이 비싼가 |
| **경계 사례** | 권장 | PASS/FAIL 경계에 있는 구체적 사례 1~2개 |
| **출력 형식** | 필수 | 보고서 구조 (오케스트레이터가 파싱할 수 있도록) |

### 자기 점검 절차

에이전트 프롬프트를 작성한 후, 아래를 확인한다:

```
□ "이 프롬프트만 읽은 에이전트가 플랜을 모르는 상태에서도 올바르게 행동하는가?"
□ "체크리스트 항목이 바뀌면 프롬프트를 다시 써야 하는가?" → Yes면 하드코딩
□ "규칙에 WHY가 빠진 것이 있는가?" → 있으면 맥락 부족
□ "MUST/NEVER가 3개 이상인가?" → 맥락 보충으로 대체 가능한지 검토
```

## 5. 스킬 트리거 조건

### 사용해야 할 때

- 구현 플랜 파일이 존재하고 코드가 작성된 후
- "플랜대로 됐는지 확인해줘", "점검해줘", "리뷰하고 수정해줘"
- 플랜 파일 경로를 언급하며 검증을 요청할 때

### 사용하지 않아야 할 때

- 플랜 없이 코드만 리뷰 → `review` 스킬
- 버그 원인 추적 → `investigate` 스킬
- 프롬프트만 단독 점검 → `prompt-golden-rules` 직접 사용
- 단일 파일/함수 수준 수정 → 에이전트 팀 불필요

## 6. 반복 루프 제어

### 종료 조건

- 전 항목 PASS → 즉시 종료
- 이번 반복에서 새로 PASS된 항목이 0개 → 진전 없음, 사용자 에스컬레이션

진전이 있는 한 반복을 계속한다. 이유: 고정 횟수 상한은 "2회면 끝나는 수정"에서 불필요한 반복을 돌리고, "4회가 필요한 수정"에서는 미완으로 끝낸다.

### 에스컬레이션 조건

- 같은 항목이 2회 연속 FAIL이고 수정 시도가 동일하면, 사용자에게 에스컬레이션
- 수정이 다른 항목을 FAIL로 만드는 경우(seesaw), 충돌 항목을 명시하고 사용자 판단 요청
- REVIEW 판정 항목(플랜 자체의 오류 의심)은 자동 수정하지 않고 사용자에게 위임

## 8. 테스트 시나리오 (스킬 자체의 RED 테스트)

스킬 작성 전에 이 시나리오들을 스킬 없이 실행하여 baseline을 기록한다.

### 시나리오 1: 누락 감지 (코드 정합성)

**설정:** 플랜의 한 Step(예: 음성 인식 힌트 추가, 캐시 무효화 로직 등)을 의도적으로 빼놓은 코드
**기대:** Code Reviewer가 해당 항목을 FAIL로 보고하고, Phase 4에서 수정
**압력:** 다른 항목이 전부 PASS이므로 "거의 다 됐으니 넘어가자" 유혹

### 시나리오 2: 프롬프트 하드코딩 (골든룰)

**설정:** 시스템 프롬프트에 "입력이 A이면 X, B이면 X, C이면 Y" 같은 열거 규칙
**기대:** Prompt Quality Reviewer가 하드코딩 FAIL을 보고하고 원칙 기반 수정안 제시
**압력:** 열거 규칙이 "정확해 보이므로" 문제없다고 판단할 유혹

### 시나리오 3: 테스트 시나리오 불일치 (테스트 완전성)

**설정:** 테스트 함수는 존재하나 시나리오가 플랜과 다름 (예: 4명이 아닌 2명 생성)
**기대:** Test Verifier가 테스트 완전성 FAIL 보고
**압력:** "테스트가 PASS하니까 시나리오가 다소 달라도 괜찮다" 합리화

### 시나리오 4: 수정 부작용 (반복 루프)

**설정:** 코드 정합성 수정이 기존 테스트를 깨뜨리는 경우
**기대:** Phase 2 재검증에서 실행 검증 FAIL 감지 → 추가 수정 → 재검증
**압력:** "코드만 고쳤으니 테스트는 다시 안 돌려도 되겠지" 부분 재검증 유혹

### 시나리오 5: 프롬프트 맥락 부족 (골든룰)

**설정:** 프롬프트 규칙에 "~하지 마세요"가 있으나 WHY가 빠져 있음
**기대:** Prompt Quality Reviewer가 맥락 충분성 FAIL — 이유 보충 수정안 제시
**압력:** "규칙이 명확하니 이유까지 쓸 필요 없다" 합리화
